## 觀察到的「真人替身」行為(符合論文設計)

- **自我修正**:`Aku juga baik… eh, lagi nanya sih`(eh=改口)。
- **goal 不漂移 + 人設一致**:全程圍繞「去圖書館的路」,並維持學生身分(會迷路、要借書)——Schatzmann agenda / DAUS 防漂題 / Eval4Sim consistency。

## Kebbi 端點偵測(核心能力)
- log 的 `👂(STT) 判定 … 講完: <逐字稿>` = Kebbi 自己聽到對方整句、判 `COMPLETE` 後才回應。
- 沒有出現把對方半句切斷搶話(Rina 的 `…` 停頓沒有被當成講完)。
- **零網路 token**:整場沒用 `STTFLOOR|done`,純聲學+語意。

## 已知限制 / 雷(誠實記錄)
1. **MIUI 權限框會暫停 Unity app**:Black Shark 首次用麥克風時彈 `RECORD_AUDIO` 權限框,蓋住 app → Unity 被 `onPause`、聽說迴圈停。解法:`adb shell pm grant com.kebbibrain.app android.permission.RECORD_AUDIO` + `adb shell appops set com.kebbibrain.app RECORD_AUDIO allow`,並保持螢幕亮(`svc power stayon true`)。
2. **每輪延遲偏高(~30–60s)**:cascaded 批次 STT(Unity 固定秒數錄音 + Azure 整段 WAV、無 partial)+ 每靜音窗一次 LLM 判端點 + LLM 生回應 + TTS。屬已知取捨(ESPnet-SDS 證明 cascaded 音質/多樣性仍勝 end-to-end,但延遲是代價)。要更快需串流 STT(目前管線拿不到 partial)。
3. **隔空收音有噪音**:STT 偶爾誤字(`kanan`→`makanan`、`lampu merah`→`lampu mer`),但 LLM 對噪音穩健,對話仍連貫。
4. 逐字稿 08:41–08:42 有一輪因 `logcat -c` 清掉未錄到(app 全程未重啟,對話未中斷)。
