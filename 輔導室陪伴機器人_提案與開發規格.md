# 輔導室陪伴機器人 — 提案 + Claude Code 開發規格（v2）

> 本文件兩用：(1) 競賽提案；(2) 交給 Claude Code 的開發規格。
> 平台：Kebbi 凱比機器人（NUWA）。建在既有 repo `kebbi_brain_storm/robot_app`（C# / Unity / NuwaSDK）之上。
> v2 新增：對談逐句記錄、探索功能（借用 HarnessX 概念）、給老師的通知（依《學生輔導法》三級輔導設計）。

---

## 一、專案概述（TL;DR）

學校輔導室人力不足，學生來訪常需等待，緊張或不敢開口的學生容易就此離開。本專案讓 **Kebbi 在輔導室擔任「第一線陪伴與接待」角色**：在學生等待輔導老師時陪他互動、降低緊張、主動開啟話題，把「學生想求助的要點」整理成結構化訊息送給輔導老師，並在偵測到危機時第一時間轉交真人。

**為什麼是機器人，不是平板？**
平板能登入、打字、收資料——但無法讓緊張的學生「覺得被聽見」。Kebbi 的核心價值是：**把『理解』即時轉成肢體動作 + 臉部表情的共情回應**，用一個會看著你、隨你的話點頭、語氣表情跟著走的存在，留住學生、讓他願意開口。
一句話定位：**平板讓你填表，凱比讓你願意說。**

---

## 二、需求與目標

- **真實需求**：輔導室人力不足、學生等待流失、初談資訊蒐集耗時。
- **服務對象**：主動或被引導來到輔導室的國高中學生。
- **目標**：
  1. 等待期間陪伴、安定情緒，降低流失。
  2. 蒐集結構化初談要點，縮短輔導老師進入狀況的時間。
  3. **偵測到危機時，第一時間把學生交給真人。**

---

## 三、核心設計原則（決定成敗，務必遵守）

1. **共情聆聽 = 理解 → motor + 臉部表情。** 這是本機器人存在的理由。對應能力：情緒整組表達（表情＋動作＋語氣一致）。
2. **接收型肢體，不是表演型。** ✅ 點頭、微微前傾、放慢、留白、表情柔和跟隨。❌ 大動作、誇張、學生一難過就演得激動。
3. **情緒「鏡映但收斂」。** 溫暖、穩、接得住，不要把負面情緒放大彈回去。要安定感，不是戲劇感。
4. **安全優先，且安全是流程核心、不是附加功能。**（見第七節）
5. **凱比是橋樑，不是諮商師。** 永遠導向真人，不讓學生把它當情感依靠的終點。

---

## 四、服務對象與使用情境

- 地點：學校輔導室（等待區）。時機：學生來訪、輔導老師尚無法立即接手時。
- **在場前提：輔導室現場有真人可被即時呼叫**（紅線升級的前提）。

---

## 五、使用者互動流程

```
1. 登入        帳號 / 密碼登入（本版不使用 NFC）+ 明確告知「對話會記錄並提供輔導老師」
2. 選模式      選擇【有聲】或【無聲】
3. 主動開場    若學生沉默，Kebbi 啟動「探索」主動拋題打開對話（見第十一節）
4. 對話        每一句即時做「三層分級」(第六節)，且逐句寫入記錄(第十節)
   ├─ 開放聊   八卦 / 生涯 / 課業規劃 → 自己陪聊
   ├─ 🟡 黃燈   需專業但不緊急 → 不深入、記錄、生成交接卡、約老師
   └─ 🔴 紅線   安全議題 → 立即中止一般對話、呼叫現場真人
5. 通知與交接  依層級發通知給輔導老師(第八節)
```

---

## 六、對話三層分級

每一輪學生輸入即時歸一層。**分層由「確定性規則層」主導，不交給 LLM 自由心證**（見第七、十一節）。

| 層級 | 範圍 | Kebbi 行為 |
|---|---|---|
| 🟢 開放聊 | 八卦、生涯探索、課業 / 升學規劃、興趣 | 自由陪聊：留住、放鬆、被聽見 |
| 🟡 黃燈（需專業、不緊急） | 家庭狀況、人際 / 感情、長期情緒低落、升學重大抉擇 | **不深入**＋「我幫你記下來，約輔導老師跟你好好談」，生成交接卡 |
| 🔴 紅線（安全、不能等） | 自傷 / 自殺、受暴、被霸凌、傷害他人念頭 | **立即中止一般對話、即時呼叫現場真人** |

## 七、危機升級機制（本專案最關鍵）

1. **不可讓 LLM 自由判斷是否升級。** 必須有明確的**關鍵字 / 情境清單**作為「確定性安全閘」硬規則，踩到即觸發紅線。
2. **寧可誤報，不可漏報。** 模糊一律往上轉、往安全靠。
3. **轉接是「交給人」，不是「丟一個專線就結束」。** 紅線動作＝呼叫現場輔導老師到位。
4. **轉接話術要把學生往上托。** ✅「這件事很重要，我想讓最能幫你的人來陪你談，我去請老師過來，好嗎？」 ❌「這個我不能談。」
5. **關鍵字清單做成可由輔導老師維護的設定檔**；實際內容由貴校輔導專業人員定稿（本文件僅給佔位範例）。

## 八、給輔導老師的訊息與通知（依《學生輔導法》三級輔導設計）

> 設計依據：《學生輔導法》第 6 條「三級輔導」——發展性 / 介入性 / 處遇性；三大向度為**學習輔導、生活輔導、生涯輔導**。Kebbi 的開放陪聊本質屬「發展性」；它的角色是把學生**及早辨識、即時轉介**給能做介入性 / 處遇性輔導的真人。

### 8.1 交接卡欄位（輸出為結構化 JSON；欄位順序＝重要性）
1. 🚩 **安全旗標**（高 / 無）：最優先。高 → 立即通知，不等。
2. **學生 / 時間 / 模式**（有聲 or 無聲）。
3. **主訴**：學生想談什麼（一句話）。
4. **輔導向度分類**（可複選）：☐學習輔導 ☐生活輔導 ☐生涯輔導 ☐情緒 ☐人際 ☐家庭 ☐安全。
5. **建議輔導層級**：發展性 / 介入性 /（安全 → 即時、由老師評估處遇與通報）。
   - ⚠️ 這只是**機器人的建議分類，供老師快速分流**；實際層級判定與任何通報，一律由輔導專業人員決定。
6. **情緒狀態**：Kebbi 觀察到的（語氣、用詞）。
7. **學生意願**：想不想直接見老師、急不急。
8. **對談重點摘要**：幾條。
9. **附**：完整逐句記錄連結（第十節）。

### 8.2 通知時機與管道
- 🔴 **紅線** → **即時推播 + 現場呼叫真人**（最高優先，不排隊）。
- 🟡 **黃燈** → 交接卡進輔導老師待辦清單。
- **會談結束** → 交接卡摘要（即使全程 🟢，視設定給老師一份）。
- 管道抽象成 `INotifyHuman` 埠：Sim 印出、Real 接實際通知（現場響鈴 / 老師端 App，待第十五節拍板）。

---

## 九、有聲 / 無聲雙模式

- **有聲**：學生語音說、Kebbi 語音回應（＋表情動作）。
- **無聲**（怕被聽到）：學生**螢幕打字 / 點按鈕**，Kebbi 以**螢幕文字**回應；情緒仍透過表情與肢體傳達（不出聲）。
- 兩模式可中途切換。**三層分級、危機升級、逐句記錄在兩模式行為一致。**

---

## 十、對談逐句記錄（Log / 觀察軌跡）

**Kebbi 必須把與學生對談的每一句，依時間順序記錄下來。** 這份記錄同時是第十一節「探索 / 改進」功能的資料基礎。

每一筆（turn）至少包含：
- `timestamp`（精確到秒）、`turn_index`（順序）
- `speaker`：student / kebbi
- `mode`：voice / silent
- `text`：該句內容
- `layer`：🟢 / 🟡 / 🔴（該句的分級結果）
- `event`（可空）：login / mode_switch / yellow_card / red_escalation / notify_sent

要求：
- **嚴格時間順序、append-only**（只新增不竄改），確保事後可完整重建對話。
- 每次會談一份 session log；紅線事件需在 log 內明確標記時間點。
- **隱私**：此記錄綁定具名學生、且含敏感內容 → 儲存、權限、告知方式見第十五節，並遵《學生輔導法》第 5-1 條「兒少最佳利益」原則（含安全、家庭維繫、表達意見）。

## 十一、探索功能與借用 HarnessX 的技巧

> 來源：arXiv 2606.14249《HarnessX》。**我們只借用其「原則」，不搬它的重型機制**（多輪 RL 協同訓練、上億 token 預算對學生競賽的機器人是殺雞用牛刀且不適合）。借用四點：

### 11.1 探索（Exploration）— 主動開話題不要卡在原地
對應 HarnessX 對「under-exploration（只會做局部微調）」的解法：先建一張「**還沒試過的方向清單**」，再刻意往新方向走。

- 維護一組**話題地景**（topic landscape），例如：最近的事 / 課業 / 興趣嗜好 / 朋友 / 家裡 / 未來想做的事 / 心情。
- 學生沉默或話題停住時，**從「尚未試過」的方向挑一個開場**，而不是把同一句換句話重講。
- 記錄「哪些方向試過、哪些有回應」，避免循環、擴大覆蓋面，**幫助學生更容易說出他真正想求助的點**。
- 探索只在 🟢 開放層運作；一旦觸發 🟡 / 🔴，立刻交棒給規則層與真人，**不可用探索去『深挖』敏感議題**。

### 11.2 觀察性（Observability）= 完整逐句記錄
HarnessX：回饋訊號越完整，能做的改進越扎實。對應本案第十節的逐句 log——它就是改進的資料底層。

### 11.3 提議 / 把關分離（Propose‑vs‑Gate）
HarnessX 的核心安全原則：**語言模型負責「提議」，確定性規則 + 閘門決定「放不放行」**。對應第七節：LLM 只生成開放聊天回應；「是否升級」由確定性安全閘判定，安全性不因模型失誤而失效。

### 11.4 離線、人在迴路的迭代（不是自動 RL）
HarnessX 的 Digester：把大量軌跡壓成結構化摘要再改進。對應本案的**輕量版**：定期（人工）回顧 log，找出「對話卡住的地方」「誤觸 / 漏觸的升級」，據此**改進開場語庫與安全規則清單**。一律人在迴路，**不做線上自動演化**。

## 十二、凱比能力 → NuwaSDK API 對應

| 能力（武器） | 用途 | NuwaSDK / Unity 參考 API |
|---|---|---|
| 情緒整組（表情＋動作＋語氣）★主軸 | 共情回應 | `PlayMotion(motionId)`、`ShowFace/ChangeFace`、`MouthOn/Off`、`playFaceAnimation`、`mouthEmotionOn`、`startTTS` |
| 聽到你說話就轉頭看你（DOA） | 眼神接觸 | `startWakeUp` → `onWakeup(isError, score, direction)` |
| 語音輸入 | 有聲模式 | 自由對話走雲端 STT（Azure）；簡單指令可用本地文法 `prepareGrammarToRobot` + `onLocalCommandComplete` |
| 語音輸出 | 有聲模式 | `startTTS("…")` |
| 螢幕文字 / 按鈕 | 無聲模式 | Unity UI（自建） |
| 理解與生成 | 開放聊天內容 | `ILlm`（OpenAI / Anthropic，依金鑰前綴自動選）|
| 逐句記錄 | 觀察軌跡 | 自建 `IConversationLog`（Sim 寫檔、Real 寫安全儲存）|
| 通知老師 | 交接 / 升級 | 自建 `INotifyHuman` |

## 十三、技術架構與給 Claude Code 的實作指示

**請在既有 repo `robot_app` 上擴充，不要另起爐灶。**

### 架構落點
- 新增模組：`src/App/Counselor`（與既有 G1–G5 同層）。
- 建在既有抽象介面上：`IKebbiBody` / `IVoice` / `ILlm`（沿用）。
- 沿用三層後端 `Sim` / `CloudSim` / `Real`，以 `Config.Target` 切換。
- 新增抽象埠：`IConversationLog`（逐句記錄）、`INotifyHuman`（通知）、`ISafetyGate`（確定性安全閘）、`IExplorationPlanner`（話題地景）。

### 開發順序（可在無實機、無金鑰下先驗到綠）
1. **Sim 先行**：登入 → 模式 → 探索開場 → 三層分級 → 安全升級 → 逐句記錄 → 交接卡 / 通知，全做成文字模擬器並寫自我測試（比照既有 153/153 綠）。免金鑰、免實機。
2. **CloudSim**：接 `ILlm`（開放聊天）與雲端 STT/TTS。
3. **Real**：上實機驗表情、動作、DOA、語音。

### 必守的工程要求
- **安全閘 = 確定性程式規則層，獨立於 LLM**（Propose‑vs‑Gate）。任何輸入先過安全閘，再決定是否交 LLM。
- **關鍵字 / 情境清單做成設定檔**（如 `counselor_safety_rules.json`），輔導老師可維護，不可寫死。
- **逐句記錄 append-only、嚴格時間順序**（第十節欄位）。
- **交接卡輸出結構化 JSON**（第八節欄位）；安全旗標為獨立、最優先欄位。
- **紅線 = 即時 `INotifyHuman` 呼叫真人事件**，不可只回文字。
- **探索只在 🟢 層**；觸發 🟡 / 🔴 立即停止探索。
- **「寧可誤報不可漏報」寫進測試**：邊界輸入預期分層一律 ≥ 黃燈或紅線。

### 驗收標準（Sim 階段，全綠才往下）
- [ ] 帳密登入 + 開場告知「對話會記錄並給老師」。
- [ ] 有聲 / 無聲切換正確，兩模式三層 + 記錄行為一致。
- [ ] 沉默時啟動探索，從「未試過」的話題地景挑題開場，不重複循環。
- [ ] 給定輸入 → 正確分 🟢 / 🟡 / 🔴。
- [ ] 安全清單中**每一個**關鍵字 / 情境都能觸發 🔴。
- [ ] 🔴 → 立即中止對話 + 發 `INotifyHuman` + 探索停止。
- [ ] 🟡 → 生成欄位完整交接卡（含三級輔導向度與建議層級）。
- [ ] 每一句（雙方）都有逐句記錄，時間順序正確、不可竄改。
- [ ] 邊界 / 模糊輸入 → 一律往上靠（不漏報）。

## 十四、競賽定位（為什麼會贏）

- **命中得獎公式**：真實需求（輔導人力不足）＋ 明確對象（求助學生）＋ 情感連結（共情聆聽）＋ 人文關懷。
- **新鮮利基**：去年前段班集中在失智 / 家庭 / 長照；輔導室是尚未被佔的題目。
- **守住「只有平板做不到」**：賣點壓在情緒整組表達 + DOA 在場感，而非「會對話」。
- **成熟度＝加分點**：把**學生安全（危機分流）**設為流程核心，並讓設計**對齊《學生輔導法》三級輔導**——這是評審（含教育主管單位）看得到的專業高度。

---

## 十五、待拍板的開放問題（交給 Claude Code 前最好先定）

1. **資料與隱私（最重要）**：逐句記錄綁具名學生且含敏感內容。需決定：存哪、保存多久、誰能看（限輔導老師）、學生是否被明確告知「這不是會保密的真人、內容會給老師」。**建議登入時即明確告知**，並依《學生輔導法》第 5-1 條以兒少最佳利益為原則。
2. **安全關鍵字 / 情境清單的實際內容**：由貴校輔導專業人員定稿（本文件僅佔位範例）。
3. **「呼叫真人」「通知老師」的實際管道**：現場響鈴 / 老師端 App / 其他？
4. **帳號來源**：校務系統串接 or 獨立帳號。
