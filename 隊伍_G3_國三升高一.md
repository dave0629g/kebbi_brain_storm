## 2. 鐵則：做出「平板做不到」的東西（你的作品命脈）
平板/手機本身就有觸控、攝影機、麥克風、喇叭、網路、TTS、ASR——所以「辨識物件→語音回饋」「問答」「朗讀」「測驗」這類平板都能做，**拿不到創意與差異化分數**。
Kebbi 真正贏過平板、平板做不到的只有四件事，**你的主題就是靠這個**：
1. **移動**（前進/後退/旋轉）：在實體空間走位、靠近人、帶路、巡邏、佈置闖關。
2. **關節手勢**（頭/雙手）：比手勢、肢體示範、指向實物、身體語言。
3. **聲源定位 + 轉向**（方向性麥克風）：知道誰在說、在哪個方位，轉頭面對說話者。
4. **多機協作**（2 台以上 Kebbi）：多台在同一空間分工、對話接力、交棒走位——**最強的平板免疫**。

> 自我檢查：把你作品裡的「移動／關節／聲源定位／多機」拿掉後，如果平板還能達到一樣的成效，就代表不夠硬，要再往身體與空間靠。

#### 〔智慧教學〕Kebbi 體育課:動作鏡像教練帶你練分解動作

**對象與發展適配(為何適合這組年齡與程式深度)**
- 國三升高一(14-15 歲)正值具體運算邁向形式運算成熟期,能把一個連續動作「拆解」成關節序列、停留秒數與節拍,這種「動作 → 結構化資料」的抽象建模正是本主題的程式核心,難度與年齡相符。
- 程式深度落在中階:學生用 Python 以資料結構(list/dict)定義動作幀(關節角度、停留秒數、節拍 BPM),寫迴圈逐拍播放、用條件判斷處理「再一次/太快了」的節奏調整,並把聲源定位回傳的方位對應到轉向角度。不需深奧演算法,但需要清楚的資料設計與狀態流程,具備可延展性(可加表、加感測回饋)。
- 體育/健康課的真實課堂情境,讓這組學生同時練到「跨科應用」與「以使用者為中心的設計」,符合智慧教學類強調的教學成效導向。

**真實需求/使用者故事**
- 體育或健康課要教標準運動動作(暖身操上肢分解、太極上肢起式、CPR 按壓的手臂與上半身姿勢),老師示範一次後就要顧全班,無法逐生對齊角度與節奏。
- 學生看影片只是被動模仿:影片是固定機位的平面影像,左右容易混淆、看不到立體角度、節奏由影片決定無法即時喊停。學生想問「這個角度對不對」時,沒有一個會「轉頭看著我」回應的帶領者。
- 需求本質:要一個站在隊伍前、面向學生做鏡像帶領、可逐拍停留、能轉向發問學生回應節奏的立體示範者。

**Kebbi 功能組合(標出平板沒有的能力)**
- 需要 Kebbi 台數:**1 台**。
- 〔平板沒有・關節手勢〕Kebbi 用**頭部與雙臂的實體關節**做出分解動作示範:逐拍擺出上肢動作幀、在每幀停留設定秒數,提供可從側面/斜後方繞看的**立體肢體參照**,而非平面影片。
- 〔平板沒有・聲源定位+轉向〕學生喊「再一次/太快了」或舉手發問時,Kebbi 以**方向性麥克風判斷聲源方位並轉頭面向該生**,做出「看著你」的面對面回應與節奏調整(放慢 BPM、回退一幀重做)。
- 〔平板沒有・實體在場〕一個會比動作、會轉頭面向你的實體教練站在隊伍前,帶來螢幕無法取代的鏡像帶領臨場感與社交專注。
- 〔平板也有,非特色〕TTS 喊口令、ASR 聽口令屬平板既有能力,在本作品僅為輔助,核心機制是上述「關節示範 + 聲源轉向」,拿掉即垮。
- 學生端 Python:定義動作關節序列與節拍資料、編排分解步驟與停留秒數、把聲源方位映射為轉向角度。

**運作流程(分步驟,具體可實作)**
1. 學生用 Python 以 dict/list 定義一組動作:每一幀含「左臂角度、右臂角度、頭部朝向、停留秒數」,並設定整組節拍 BPM 與循環次數。例:暖身上肢開合 6 幀。
2. Kebbi 站在隊伍前、面向學生,先語音預告「鏡像帶領,跟我的左手就是你的右手」。
3. Kebbi 依節拍**逐幀擺出上肢與頭部動作**,每幀停留設定秒數,口令報幀(「第二拍,手臂平舉」)。
4. 學生跟做。任一學生喊「太快了」→ Kebbi **聲源定位該方位並轉頭面向**,回應「好,放慢」並把 BPM 下調一級;喊「再一次」→ 回退一幀重來。
5. 完成一輪後,Kebbi 語音切換下一組動作,重複 2-4。
6. (延伸)記錄本輪被喊停的幀次,做為下次起始節奏建議。

**為何平板做不到-核心論證**
- 核心機制是**「真實關節實體做出可繞看的鏡像示範」+「聲源定位轉頭面向發問者」**,兩者皆非平板能力:平板是固定平面螢幕,影片需做左右翻轉才不混淆鏡像,且無法讓學生繞到側面/斜後觀察動作的角度與方向;3D avatar 仍是平面投影,對空間感弱的學生缺乏立體實體參照。
- 「轉頭看著發問學生回應並當場調節奏」是物理動作,平板只能螢幕點名或全場喇叭回應,沒有那種被機器人轉頭凝視的帶領臨場感與社交專注效果。
- 誠實面對的邊界(並已設計規避):Kebbi 僅頭與雙臂 2 自由度,做不出下肢、全身轉體與跪姿按壓深度,因此本作品**刻意把示範範圍鎖定在「上肢/頭部可做的分解動作」**(暖身上肢開合、太極上肢起式、CPR 的肘直手臂姿勢角度),讓核心機制落在 Kebbi 物理優勢之內,不誇稱全身運動。如此論證即無懈可擊:在「上肢鏡像 + 轉向帶領」這個被選定的範圍內,平板確實做不到立體繞看與轉頭面對。

**對應評分亮點**
- 應用性:對準真實體育/健康課的標準動作教學痛點,單台即可進班,情境具體。
- 創意性:把「動作」抽象成關節幀+節拍資料,結合聲源轉向做動態節奏調整,跨科融合。
- 挑戰性:聲源方位→轉向角度映射、逐幀節拍同步、即時節奏狀態調整,具實作難度。
- 完整性:從學生編資料 → Kebbi 帶領 → 聲源回饋 → 切換動作,流程閉環可重複。
- 展演:鏡像帶領 + 轉頭回應發問者,現場觀感強、互動清楚。

**挑戰度與程式深度**
- 挑戰度:★★★☆☆(3)。
- 程式深度:中階。關節幀資料結構設計、節拍迴圈與停留控制、聲源方位到轉向角度的映射、節奏狀態機(放慢/回退/切換)。不含複雜演算法,但需穩健的資料與流程設計。

**可現場 demo 的 MVP(先做哪一條最小流程)**
- 最小流程:一組 4-6 幀的「上肢暖身開合」,Kebbi 面向學生逐拍鏡像帶領,並對「太快了」這一句做聲源轉向 + 放慢一級 BPM。先把「關節逐幀示範 + 一個聲源轉向回應」跑通即可證明核心機制,動作切換與多組資料為後續加分。

**展演高光時刻(6 分鐘最吸睛橋段)**
- 高光:Kebbi 正在逐拍帶暖身上肢動作,左後方一名學生大喊「太快了!」,Kebbi 當場**轉頭面向那名學生**說「好,放慢」,並把節拍明顯放慢、回退一幀重帶——全場看到機器人「聽見方位、轉頭看你、即時調整」的鏡像帶領臨場感,正是平板給不出的畫面。

#### 〔互動遊戲〕迷宮特派員:雙機合作的盲眼導航默契挑戰

**對象與發展適配（為何適合這組年齡與程式深度）**
國三升高一（14-15 歲）正處於抽象思考與系統設計能力快速成熟的階段，能駕馭多角色、多狀態、多機通訊的程式架構，也能承擔「精準口語溝通」這種需要後設認知與情緒控管的團隊任務。本作品在程式上需要兩台 Kebbi 跨機 API 同步任務狀態、計分與計時，並整合移動序列、聲源定位判斷、手勢觸發三條獨立邏輯線，難度恰好落在「會卡住但做得出來」的甜蜜點，避免太簡單而流於紙上談兵，也不至於超過此年齡層的負荷。對團隊合作課而言，盲眼導航的張力（指揮者看得到、執行機看不到）天然製造溝通落差，正是 14-15 歲學生練習「把腦中地圖轉成對方聽得懂的精準指令」的最佳載體。

**真實需求/使用者故事**
團隊合作課老師明確表示「想訓練學生精準口語溝通與信任，要一個需要兩人一機協作、靠移動才能完成的實體任務，而非紙上談兵」。傳統的信任遊戲（蒙眼牽手、背對倒下）缺乏可量化的回饋與重玩價值；本作品讓「指令是否精準」直接反映在機器人是否撞線、用了幾條指令通關，把抽象的溝通品質變成看得見、可比較、可復盤的具身結果，滿足老師對「真實、可評量、需移動」的需求。

**Kebbi 功能組合（標出平板沒有的能力）｜需要 2 台**
- 【移動・平板沒有】特派 Kebbi（甲機）在地面膠帶迷宮中真實前進/後退/旋轉，依語音指令穿越實體路徑。這是平板根本做不到的物理位移。
- 【聲源定位＋轉向・平板沒有】甲機靠方向性麥克風判斷「是指揮者 A 的方位」並轉身面向他，確認當前指令來源，弱化旁人干擾、強化「信任特定隊友」的方向感。
- 【移動＋關節手勢・平板沒有】裁判 Kebbi（乙機）自走巡場，偵測到撞線時主動移動靠近犯規點，並用手臂比出「扣分」手勢，是會走會比手勢的獨立物理代理。
- 【多機協作・最強平板免疫】甲乙兩機透過 API 即時同步任務進度、計時與計分：甲機回報自身座標/移動步數，乙機回報撞線判定，雙方共構同一場比賽狀態。兩台會移動會比手勢的實體智能體在同一塊地面分工，連多台平板也無法取代其移動與物理臨場。
- 平板已具備的（觸控、攝影機、麥克風、TTS/ASR）僅作輔助 UI（指揮者地圖、計分板顯示），不構成核心機制。

**運作流程（分步驟，具體可實作）**
1. 佈置場地：用膠帶在地面貼出迷宮牆線與起點/終點，指揮者 A 拿到對應的紙本/平板地圖（執行機看不到地圖）。
2. 特派甲機停在起點，乙機停在巡場待命位，兩機 API 握手、初始化計分=0、計步=0、計時啟動。
3. 指揮者 A 用語音下達原子指令（「前進兩步」「左轉」「停」）；甲機先以聲源定位轉頭面向 A 確認指令來源，再執行對應移動。
4. 每執行一條指令，甲機指令計數 +1 並回傳至共享狀態。
5. 乙機沿外圈/內巡路線移動巡場，持續比對甲機座標與膠帶牆線；若判定撞線，乙機移動靠近該點、舉手比「扣分」手勢，並透過 API 將分數扣除、語音播報。
6. 甲機抵達終點 → API 標記任務完成、停止計時，平板/語音公布「指令數 / 撞線數 / 耗時」。
7. 換隊重玩，最後比較哪隊用最少指令、最少撞線通關，進行溝通復盤。

**為何平板做不到-核心論證（差異化關鍵）**
本作品的學習成效本體不是「抽象溝通」，而是「指揮者看著真實場地、被導航的機器人在同一塊地面上真的走位、用身體穿越膠帶迷宮、撞到真牆會被裁判機器人移動過來比手勢判定」的共享物理空間體驗，平板有三點根本無法等效：
1. 被導航者必須是物理地面上真會走位、真會撞牆的實體——平板只能呈現螢幕/AR 虛擬格子，失去「真的會撞、真的要走位」的身體張力與信任感；老師要求的正是「靠移動完成、而非紙上談兵」。
2. 裁判角色是會自己移動巡場、用手臂比手勢的獨立物理代理——定點、無臂的平板無法在同一空間移動巡邏與肢體判定，只能用 App 數字提示取代，喪失「裁判走過來比手勢扣分」的臨場張力。
3. 聲源定位轉身面向特定指揮者，是「機器人用身體回應你」的物理回饋——平板只能用喚醒詞/按鈕模擬，弱化「信任特定隊友方向」的核心設計。
拿掉移動、聲源定位轉向、雙機物理巡場與手勢判定，這個任務就整個垮掉。多台只能定點、無手臂、不能移動的平板無法達成兩機在同一地面移動＋肢體協作的物理本體，故成效會明顯改變。可被平板取代＝false。

**對應評分亮點**
- 應用性：直接命中團隊合作課「精準溝通與信任」需求，把抽象溝通量化為指令數/撞線數，可復盤、可重玩。
- 創意性：盲眼導航＋聲源定位「認指揮者」＋會走來比手勢的裁判機，三層機制疊加，少見且新穎。
- 挑戰性：跨機 API 即時同步、移動序列、聲源定位、手勢觸發四線整合（★4）。
- 完整性：起點→導航→巡場判定→終點→計分→換隊復盤，閉環完整且狀態同步清楚。
- 展演：兩台機器人同場移動、轉頭認人、舉手扣分，視覺與戲劇張力強，現場觀眾一看就懂。

**挑戰度與程式深度**
挑戰度 ★4。程式深度：需實作（1）甲機語音指令解析→移動序列；（2）甲機聲源定位轉向確認來源；（3）乙機自走巡場路徑＋撞線判定邏輯；（4）乙機手勢動作觸發；（5）雙機 API 即時同步進度/計時/計分的狀態機。屬中高階整合專案，重點在多機通訊與狀態一致性。

**可現場 demo 的 MVP（先做哪一條最小流程）**
最小流程：單台甲機在簡化直線+一個轉彎的膠帶路徑上，接收「前進兩步、左轉、前進一步」語音指令，先以聲源定位轉頭面向指揮者再移動，抵達終點語音回報指令數。先打通「語音→轉向確認→移動→計數」這一條，確認移動與聲源定位穩定後，再接入乙機巡場與雙機 API 同步。

**展演高光時刻（現場 6 分鐘最吸睛橋段）**
高光在第 3-4 分鐘：指揮者喊「左轉、前進兩步」，甲機先轉頭面向指揮者（觀眾看到機器人「認人」），再真的在地面迷宮走位；就在甲機壓到膠帶牆線的瞬間，裁判乙機自己移動巡到犯規點、停下、緩緩舉起手臂比出「扣分」手勢並語音播報——兩台機器人同場一走一判、一轉頭一比手勢，這個「會動的裁判走過來抓包」的畫面是全場最戲劇性、平板絕對做不出的一刻。

#### 〔生活應用〕情緒值日機器人:會主動靠近沉默成員的家庭關懷哨兵

**對象與發展適配**
- 適合國三升高一(14-15 歲):此年齡層正處於情緒起伏大、習慣「沉默封閉」而非主動求助的階段,做這個題目能讓學生把切身的情感經驗轉化為設計動機,投入度高、論述真誠。
- 程式深度剛好落在「進階但可達成」:需要狀態機(巡邏/接近/陪伴/通報)、感測資料判讀(聲源方位、音量、靜默時長)、Python 串接通訊 API(LINE Notify / Telegram Bot)、動作序列編排(移動+轉頭+手勢),涵蓋條件判斷、計時器、事件驅動、API 串接,對升高一學生是有挑戰但能完成的甜蜜點。
- 同時練到「軟性議題硬技術化」的設計思維,符合生活應用類重視真實情境與同理心的評分取向。

**真實需求/使用者故事**
- 家中青少年或長輩心情低落時常一個人不出聲,家人忙於工作家事沒能及時察覺,等發現時往往已累積一段時間。
- 使用者希望有一個夥伴能「主動巡邏發現」沉默獨處的成員、溫柔走近開啟對話、陪伴傾聽,必要時在徵得同意後通知家人來關心 — 把「被動等人開口」變成「主動被看見」。

**Kebbi 功能組合(標出平板沒有的能力,需 1 台)**
- 需要 Kebbi 台數:**1 台**(單一夥伴設計,刻意不做成多機監控網,符合「家庭夥伴」而非「監視系統」的定位)。
- **[移動 — 平板沒有]** 定時在客廳/餐廳/房間之間巡邏走位,以單台覆蓋全家空間;偵測到目標後主動前進、靠近到陪伴距離。
- **[聲源定位+轉向 — 平板沒有]** 用方向性麥克風判斷哪個房間有人但長時間安靜或聲音低沉,並轉頭/轉身面向說話者,做出「我在聽你說」的具身回應。
- **[關節手勢 — 平板沒有]** 放慢動作轉頭面向對方、用柔和手勢比出「要不要聊聊」、傾聽時點頭回應、願意時帶一段簡單放鬆律動。
- **[實體在場 — 平板沒有]** 上述綜合而成:一個會走過來、會轉頭看你、會點頭的實體夥伴,傳遞「有人特地為你而來」的被在乎感。
- **(平板也有,不算特色)** 麥克風收音、TTS 語音對話、ASR 語音辨識、網路串接 API — 這些只是輔助管道,不是核心機制。

**運作流程(分步驟,具體可實作)**
1. **排程巡邏**:依設定時段(如放學後、睡前),Kebbi 以預設路徑在客廳/房間間移動巡邏,覆蓋全家空間。
2. **靜默偵測**:每到一個空間,用聲源定位+音量分析判斷「此處有人(偵測到呼吸/翻身/低語等微弱聲源)但長時間無對話或聲音低沉」,以靜默時長 + 低音量 + 獨處作為「可能情緒低落」的觸發條件。
3. **主動接近**:符合條件時,Kebbi 放慢移動速度走近到約 1 公尺的陪伴距離,**轉頭面向對方**,避免突兀。
4. **溫柔破冰**:用柔和手勢 + 低音量 TTS 開場:「我剛好經過,想陪你一下,要不要聊聊?」
5. **陪伴傾聽**:對方說話時,以聲源定位持續面向他、**點頭回應**;對方不想說也不強迫,改為安靜陪伴一小段時間。
6. **互動升級(可選)**:對方願意則一起做一段放鬆律動(深呼吸引導 + 簡單手勢動作)。
7. **徵得同意通報**:若多次巡邏皆偵測到持續低落,Kebbi 先口頭詢問「要不要我請家人來陪你?」,**取得同意後**用 Python 串接 LINE Notify / Telegram Bot 發關懷訊息給家人,避免變成監控。

**為何平板做不到 — 核心論證(無懈可擊)**
- **(1) 巡邏與覆蓋**:平板固定一處,要感知全家需在每個房間各放一台,仍有死角(廁所、陽台、未裝平板處),且多台等於監控網,偏離單一夥伴設計;單台 Kebbi 用移動即可巡行覆蓋全家。
- **(2) 主動接近的物理在場**:對封閉、低落的人而言,「有人特地走過來、放慢動作、轉頭面向我」本身就是被在乎的訊號;固定平板只能原地發聲亮螢幕,缺少「向我移動」這個身體語言,破冰力量明顯較弱。
- **(3) 轉頭面向 + 點頭傾聽的具身回應**:平板無關節、不能轉頭朝向人,傾聽姿態只能是螢幕動畫,溫度感受截然不同。
- **(4) 主動發現沉默者**:平板/音箱是被動等聲音觸發,而本作要解的正是「沉默到沒人察覺」這種低訊號情境 — 無聲時固定裝置幾乎收不到觸發,唯有會主動巡行的實體能去「找」一個刻意安靜的人。
- 結論:拿掉移動與實體在場,「主動發現沉默成員」與「肢體陪伴破冰」這兩個核心成效會實質塌陷,而非僅是體驗差異。平板版只會退化成一個要人自己點開的 App。

**對應評分亮點**
- **應用性**:直擊真實家庭場景(青少年/長輩情緒低落被忽略),有明確使用者與可落地的關懷+通報流程。
- **創意性**:把「巡邏哨兵」概念用在情感關懷,以「靜默」而非「聲音」作為觸發,逆向思考亮眼。
- **挑戰性**:整合聲源定位判讀、靜默偵測演算法、移動路徑、動作序列、API 串接,技術面向多元。
- **完整性**:從巡邏→偵測→接近→陪伴→通報形成閉環,並含「徵得同意」的倫理設計,考量周到。
- **展演**:會走過來、轉頭、點頭、比手勢的橋段具高度戲劇張力與情感渲染力。

**挑戰度與程式深度**
- 挑戰度:**★★★★☆ (4/5)**
- 程式深度:狀態機(巡邏/接近/陪伴/通報四態切換)、聲源方位與音量/靜默時長的感測判讀邏輯、移動路徑與避障、動作序列編排(移動+轉頭+手勢同步)、Python 串接通訊 API 與「同意後才發送」的條件控制。難點在「靜默+低落」的觸發門檻調校與避免誤判/過度打擾。

**可現場 demo 的 MVP(先做最小流程)**
- 最小可行流程:**靜默偵測 → 主動移動接近 → 轉頭面向 → 點頭傾聽 → 開口破冰**這一條主線。
- 先固定一個「房間」場景,讓 Kebbi 從起點巡邏走到偵測到「安靜的人」的位置,走近、轉頭面向、比出柔和手勢說「要不要聊聊」、傾聽時點頭。通報功能可用一支簡單 LINE Notify 串接做加分項,不列入 MVP 必要路徑。

**展演高光時刻(現場 6 分鐘最吸睛橋段)**
- 高光點:一名「演員同學」安靜坐在角落低頭不語,Kebbi 從舞台另一端**緩緩巡邏移動過去**,在接近時**放慢速度、轉頭面向他**,用柔和手勢比出「要不要聊聊?」,同學開口後 Kebbi **持續面向並點頭傾聽**,最後一起做一段放鬆律動 — 全程「移動+轉頭+點頭+手勢」連續呈現,把平板做不到的「主動走向沉默者」的情感衝擊一次釋放,評審一眼就懂為何非 Kebbi 不可。
