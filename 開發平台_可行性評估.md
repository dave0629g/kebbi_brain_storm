# Kebbi 開發平台可行性評估：CodeLab vs RaaS(流程式) vs Unity

> 重新檢視三種開發 Kebbi 的方式,對照我們五隊主題的實際需求,給出每隊建議與必須先確認的風險。
> 結論先講：**你的判斷大致正確**——Unity 最彈性、可串 LLM、功能最多元；CodeLab 進階功能難做；RaaS/流程式工具有授權(license)問題。但有一個**所有平台共通的關鍵風險(聲源定位 API)** 必須先向女媧確認。

---

## 一、三種開發方式總覽（已查證）

### A. NUWA CodeLab（積木式，no-code）
- **是什麼**：瀏覽器/App 上拖拉積木(Blockly 式)控制 Kebbi，免安裝、免寫程式。內建 36 關任務卡與分級教材。
- **能做**：內建動作(motion)、TTS/ASR、臉部、基本感測、簡單流程邏輯。
- **限制（對應你說的「很難開發」）**：
  - 只能用內建積木，**難以實作自訂演算法**（如讀 DOA 角度換算方位扇區、姿態相似度計算、複雜狀態機）。
  - **難以串接外部 LLM / 任意雲端 API**（功能被內建積木框死）。
  - **多台機器人連線需付費**：CodeLab 分 Free Trial / Home(機主) / **Pro / Team(付費)**；同時連多台機器人是付費(Pro/Team)層級才有。→ 我們的雙機/三機主題若用 CodeLab，**要買付費授權**。
- **適合**：低年級、入門、單機、邏輯不複雜的作品（如 G1）。

### B. RaaS Suite / 流程式雲端工具（你說的「Roflow/RoBoFlow」）
- **是什麼**：女媧的 **RaaS Suite（Robot as a Service）** 雲端機器人服務平台 + No-Code 雲端開發工具，內含 **GenAI** 模組。
- **授權問題（對應你說的 license 問題，已證實）**：RaaS Suite 與 GenAI 是**需要訂閱/授權**的商業平台——這次競賽的首獎獎品就是「**RaaS Suite 一年授權 + GenAI(10M/月) 一年訂閱**」，正說明它平常要付費。
  - 風險：**比賽期間若沒有授權，可能無法用 RaaS 的 GenAI / 進階雲端功能**；得獎才送一年。
- **適合**：已有授權者做內容/對話流程設計快；但綁訂閱、且彈性不如 Unity。
- ⚠️ 我查不到女媧有一個正式名稱叫「RoBoFlow/Roflow」的產品；最接近的是 **RaaS Suite 的無程式雲端工具**。請在工作坊跟女媧確認你指的是哪一個。

### C. Unity（+ 官方 Unity SDK，可寫 C# / 串 LLM）
- **是什麼**：女媧提供 **NuwaUnity SDK**（也可用 Android Studio），Kebbi 本體是 Android，可把 Unity App 部署上去。
- **已查證 SDK 提供**：✅ 底盤移動(Movement control)、✅ 馬達/關節角度控制(Motor control、角度範圍表)、✅ 臉部追蹤/辨識(FaceTrack/Face_Recognize)、✅ TTS、✅ 本地與雲端 ASR(Local ASR / Cloud Speech To Text)、✅ 攝影機、觸控、PIR、LED。
- **LLM / 外部 API（對應你說的「Unity 可用 LLM、功能更多元」——正確）**：Unity 是完整 C# 環境，**可用 `UnityWebRequest` 直接 HTTP 呼叫任何 LLM API（Claude API 等）**，不受內建積木限制 → 對話、出題、糾錯、生成情境都能自訂。**這是 Unity 對 CodeLab 的決定性優勢。**
- **多機協作**：SDK 文件未直接寫多機通訊，但 Unity 可用**標準 socket / MQTT 自行做機器間通訊**（我們各隊設計的雙機/三機協定正是這條路）。
- **代價**：需要 **Android Studio + Unity + C# 程式能力**，學習曲線高 → 適合高中（G4/G5），國中（G2/G3）需指導老師較多協助。

## 三、⚠️ 三個必須先向女媧/工作坊確認的關鍵風險

### 風險①（最重要）：聲源定位的「角度/方位」能不能用程式讀到？
- **硬體有**：Kebbi Air S 明列「6 麥克風陣列 + 聲源定位」，機器人本來就會**自動轉向說話者**。
- **但 SDK 文件查不到讀取 DOA 角度/方位的 API**。差別在：
  - 若只要「**轉頭面向說話者**」這個內建行為 → 很可能可用（G3 鏡像教練、G5 點名轉向、各題「轉向發言者」多半 OK）。
  - 若要**讀出精確角度/判斷學生在哪個方位扇區當對錯真值** → **這是 G4『Tebak Arah』與 G2『聲源捕手』、G1『誰在叫我』的命門**，必須確認 SDK 有沒有開放 DOA 數值。
- **若沒開放**的備案：改用「轉向＋攝影機定位人」混合判斷、外接麥克風陣列自行 DSP、或把題目從「讀角度」降級為「轉向面對」。
- 👉 **工作坊第一個要問女媧的問題。**

### 風險②：Kebbi 支不支援「印尼語」ASR/TTS？
- Kebbi 內建語音以中文/英文為主，**印尼語語音辨識/合成是否支援未知**。
- 若不支援：G4 用 **Unity 串接外部雲端（Google/Azure 印尼語 STT/TTS）** 經 HTTP 解決——這又是 Unity 的優勢，CodeLab 做不到。
- 👉 G4 必確認。

### 風險③：多機與 GenAI 的授權
- 多機協作：CodeLab 要 **Pro/Team 付費**；Unity 走自寫通訊(免授權但要寫)。
- GenAI/RaaS：**比賽期間是否已有授權**？得獎才送一年。若沒授權，LLM 走 **Unity + 自備 API key** 最穩。
- 👉 影響 G1/G2/G5(多機) 與所有想用 LLM 的隊。

## 四、每隊建議開發平台（對照各隊主攻主題需求）

| 隊 | 主攻主題 | 技術需求 | 建議平台 | 理由 |
|---|---|---|---|---|
| **G1** 小五(1人) | 雙機接力闖關(程式) | 移動 + 多機 + Blockly | **CodeLab** | 積木式最適合小學;教的就是程式積木本身。多機需 **CodeLab 付費層**,或先做**單機版**免費跑通。 |
| **G2** 國二 | 幾何證明接力站 | 移動到座標 + 關節指向 + 雙機同步 | **CodeLab 起步 → Unity 進階** | MVP 可用進階積木;要精確走位/雙機握手建議轉 Unity。 |
| **G3** 國三升高一 | 體育動作鏡像教練 | 逐幀關節角度 + 轉向發言者 | **Unity(建議) / CodeLab(簡版)** | 逐幀精細關節 + BPM 控制,Unity 較順;轉向用內建即可。 |
| **G4** 高一升高二 | 印尼語 Tebak Arah | **DOA 讀角度 + 印尼語 ASR/TTS + LLM** | **Unity + LLM(強烈建議)** | 需自訂方位判定、外接印尼語語音、串 LLM 出題糾錯——**正是 Unity 主場**,CodeLab 做不到。 |
| **G5** 高三升大學 | 歷史法庭辯論劇場 | 雙機通訊 + 移動 + 轉向 + LLM | **Unity + LLM(必須)** | 多機 socket/MQTT + LLM 生成辯詞 + 複雜狀態機,只有 Unity 撐得起;也最適合衝大獎與備審作品集。 |

**總原則**：
- **高中組(G4/G5)走 Unity + LLM**——和你的判斷一致,功能最多元、最能拿挑戰性與完整性分。
- **國中組(G2/G3)** 視程式力決定:先用 CodeLab 做 MVP,行有餘力升 Unity。
- **小學組(G1)用 CodeLab**(符合年齡與「教程式」本質),多機部分評估付費或做單機版。
- **盡量避免把作品命脈押在 RaaS/GenAI 授權上**(沒得獎可能沒得用);要 LLM 就用 Unity 自帶 API key 最穩。

# 八、SDK 實測重大更新（2026-06-18，來自 Unity SDK session：反編譯 aar + demo 程式 + H201 真機實測交叉驗證）

> 以下取代上文多處「未知」。**狀態圖例：✅已真機確認／⚠️API 有但被授權牆擋或未驗／❌沒有。**

## 8.1 三個改變全局的前提
1. **開發機是 NUWA AIR-H201（桌上型、無可動底盤）→ 不會實體移動。** 所有「底盤移動／走位／巡邏／靠近／接力走」命令在 H201 上不會動（只有輪式的大 Kebbi 會動）。⇒ **凡是把「移動」當核心機制的設計，在 H201 上不成立。**
2. **Active-code 授權牆**：沒授權時，**非同步硬體命令**（turnToDOA、內建 TTS、motionPlay、LED）會**靜默 no-op（不報錯）**；但**同步讀取**（馬達角、DOA）、**事件回呼**、**對外網路**免授權可用。
3. **未公開 API**：DOA 讀取、SLAM、ConnectionManager（多機）只存在於反編譯 binary，官方文件明寫「沒有」→ 可用但 Nuwa 不背書、未來可能變動。

## 8.2 驗證後的能力現況
| 能力 | 現況 | 怎麼用 / 注意 |
|---|---|---|
| **DOA 聲源角度** | ✅ 可讀（連續度數，非扇區） | `getDirectionOfDOA():float` ＋事件 `onWakeup(...,direction)`／`onFaceSpeaker(direction)`；onWakeup 已在 H201 實測；走**免授權**同步管道。⚠️解析度／正後方／非語音聲音是否更新＝未驗 |
| **關節（頭2＋雙臂8，共10馬達）** | ✅ 實測可用、免授權 | `setMotorPositionInDegree` 設角、`getMotorPresentPossitionInDegree` 讀角；**逐幀體操、手臂指向都能做**（自寫 coroutine 比內建 motionPlay 穩——後者可能被授權牆擋） |
| **頭轉向說話者** | ✅ 可自做 | 別用可能被擋的 `turnToDOA`；改「讀 DOA →自己 setMotor 把 NECK_Z 轉到該角」 |
| **網路／LLM／雲端** | ✅ 完全可用、無沙箱白名單 | UnityWebRequest 打任意 HTTPS（LLM、雲端印尼語語音）；UDP 送已 H201 實測 |
| **相機自取影像** | ⚠️ 可繞過 Nuwa | Unity `WebCamTexture` 取像素跑自己的 MediaPipe；但 Kebbi OS 是否准一般 app 開相機＝未驗 |
| **內建 TTS／語言** | ❌ 無印尼語（僅 7 語）＋TTS 可能被授權牆擋 | 改走**雲端印尼語 TTS ＋ Unity AudioSource 播放**（見 `印尼語語音_線上方案.md`） |
| **取麥克風 PCM** | ⚠️ SDK 不給 | 自己用 Android `AudioRecord`；**麥克風單一擁有者**，要先 `stopListen` 釋放；**「系統持麥時 3rd-party 能否搶麥」＝上線前第一個必測** |
| **底盤移動（距離／角度／座標）** | ❌（且 H201 無底盤） | 只有連續速度 `move/turn`，無里程計、無避障、無 goto |
| **多機 Kebbi↔Kebbi** | ⚠️ 能力在、未實證 | 非公開 ConnectionManager（Wi-Fi WebSocket P2P）但 Unity glue 不完整（缺 startListen／send）；純 socket「送」已驗、「收」未驗；**從未雙機實跑** → 需補 Java glue ＋ 雙機實測 |
| **人體 pose／skeleton** | ❌ SDK 無 | 只能臉框中心推方位、臉框大小粗估距離；要骨架得自己 MediaPipe |

## 8.3 黃金 workaround 心法（繞過授權牆）
> **讀（DOA／馬達角，免授權）＋ 自己寫馬達角度（免授權）＋ 對外網路（免授權）＝ 蓋掉大多數需求；避免依賴 async 內建（turnToDOA／TTS／motionPlay／LED）。**
- 轉向說話者：讀 DOA → 自寫 NECK_Z 轉角（不用 turnToDOA）。
- 說話：雲端 TTS → Unity AudioSource（不用內建 TTS）。
- 動作：逐幀 setMotor（不用 motionPlay）。

## 8.5 上線前必做的三個真機實測（按優先序）
1. **搶麥**：系統 wakeup 持麥時，自己的 AudioRecord 能否取得 PCM（決定 G4 雲端印尼語整條管線）。
2. **相機權限**：一般 Unity app 能否用 WebCamTexture 開 Kebbi 相機（決定 G3／G4 自跑 CV）。
3. **DOA 細節＋多機**：getDirectionOfDOA 解析度／360°；（若要多機）雙機 socket 收送一次。

> 結論修正：硬體現實**強烈偏好「關節＋DOA＋雲端語音＋LLM、單機」設計（G3、G4）**；**移動／多機設計（G1、G2、G5、彩蛋）需視是否有輪式 Kebbi 與多台而定**。
