## 1. SDK 核對結果(NUWA Robot SDK ver 1.1.5 → Air S)— 影響設計的重點

已核對**新版 Javadoc**(`NuwaRobotAPI`,**SDK 2.1.0.08** = 我們一直用的 aar 版本)+ 官方 `NuwaSDKExample`(涵蓋 Kebbi / Danny / **Kebbi Air**)。**以新 Javadoc 為準**(舊 1.1.5 HTML 已過時,先前有一處判斷錯已更正):

1. **❗ 沒有 DOA(聲源方向)API**(新舊版都確認):只有語音喚醒(「Hello Kebbi」→`onWakeup()`),沒有任何聲源方向。我們碼裡 `ReadDoaDegrees()` **無對應**。
   - 輔導場景剛好不痛:**桌上型、學生坐正前方 → 面向前方即可**,本來就不需要靠聲音找方向。EmpathyBody 的「眼神接觸」改成「面向前方 + 點頭/前傾」。
   - (只有印尼語方位遊戲 TebakArah 真的依賴 DOA,但它不在旗艦路線。)
2. **✅ 搶麥可行**:`stopListen()` 存在;可用 meta-data `disableAlwaysWakeup=true` 關常駐喚醒避免它佔麥 → **搶麥①路線確認**。
3. **✅ 觸控完整對上**:`requestSensor(SENSOR_TOUCH)` →`registerRobotEventListener`→`onTouchEvent(type,touch)`。type:頭1/胸2/右手3/左手4/左臉5/右臉6;碰1/放0。我們 `ITouchSensor`(#10)對得上。
4. **🆕 PIR 人體感測器(取代相機做「在不在」)**:`requestSensor(SENSOR_PIR)`→`onPIREvent(val)`。硬體判「有沒有人」,**非視覺、零雲端、零延遲、零隱私風險** → 正好替代相機版 SafetyEye:**學生靠近→打招呼、離開→降待命**,完全符合「不做連續視覺」。
5. **🙂 臉部表情有 `playFaceAnimation`(新 SDK 2.1.0.08 確認 — 重要更正)**:`playFaceAnimation(int id)` / `playFaceAnimation(String 名稱)` 播內建臉部動畫、`mouthEmotionOn(speed, type)` / `mouthOn(speed)` 控嘴型、`showFace()`/`hideFace()`。→ **Air S 有一張內建會動的臉,可直接依情緒(🟢🟡🔴/傾聽)播表情動畫 + 講話時動嘴**,比自繪 UI 豐富很多。(舊 1.1.5 HTML 沒這些 API、我先前說「只能自繪」是錯的;以新 Javadoc 為準。仍需上機確認授權放行。)
6. **💡 LED**:`setLedColor(id,brightness,r,g,b)`,id 1臉/2胸/3左手/4右手,需先 `disableSystemLED()`;**Air 不支援臉/胸 LED 呼吸燈**(手部 LED 可純色)。
7. **🦾 馬達**:`ctlMotor`(NeckZ ±40);Air `move/turn` 底盤移動無效。

> **Air S 的全部表達手段** = 頭(±40)+ 雙臂 + **內建臉動畫(`playFaceAnimation`)** + 手部 LED + 觸控 + PIR + 語音(Azure)。

## 1.5 整合架構分岔:CustomBehavior vs 獨立 app(待討論)

來源:Nuwa SDK Training Kit `custombehavior_arch.png`。NUWA Robot Behavior System 處理語音喚醒→ASR→Nuwa NLP,把意圖分派到「Nuwa 內建行為(Q&A/TTS)」或「你的 **CustomBehavior**(`NuwaSDK extends BaseBehavior`)」。

| | **走 CustomBehavior**(Nuwa 官方整合) | **走獨立 app**(我們現況) |
|---|---|---|
| 觸發 | 語音指令/NLP 意圖(「找輔導老師」)→ 系統 callback | 自己當 launcher / 選單按鈕 |
| 語音輸入 | **Nuwa 系統給 ASR 文字** → **可能直接繞過搶麥①** | 自接麥克風 + Azure STT + VAD(#6),要解搶麥 |
| 取捨 | 省搶麥;但綁 Nuwa ASR/NLP,**中文逐字品質要夠餵安全閘** | 全控、品質可選;但要解搶麥授權牆 |

→ **旗艦整合的關鍵決策**:輔導室陪伴要不要做成 CustomBehavior(省搶麥、官方路徑)還是獨立 app(全控 STT)。建議 P0 上機時**兩條都快速試**:CustomBehavior 拿到的中文 ASR 餵一句安全閘看判得準不準;不準再走獨立 app + Azure STT。

## 3. 建議實機優先序

### 🔴 P0 — 一次驗「Air S 授權牆放不放行」(~20 分鐘,ROI 最高)
一台 Air S、接 adb,把旗艦會用到的硬體手段一次測完:
1. **搶麥①** — `disableAlwaysWakeup` + `stopListen()` → 錄 → STT 出文字?
2. **馬達②** — `ctlMotor` 命令 NeckY 點頭、NeckZ 微轉、肩臂前傾 → 有沒有動、幅度順不順(記校準值)。
3. **觸控③** — `requestSensor(SENSOR_TOUCH)` → 摸頭看 `onTouchEvent` 有沒有來。
4. **PIR** — `requestSensor(SENSOR_PIR)` → 人走近/離開看 `onPIREvent`。
5. **臉表情⑤** — `playFaceAnimation(一個情緒)` + `mouthOn` → 臉會不會變、嘴會不會動。

→ 這 5 項決定「輔導室陪伴」的具身/感知/語音能不能用。任何一項 silent fail,該段就走降級(純語音仍可運作)。

### 🟡 P1 — 輔導室陪伴端到端 demo(P0 通過後)
跑一段真實會談:**登入告知 → 學生說話(VAD 窗)→ 安全閘 🟢/🟡/🔴 → 綠燈真 LLM 陪聊 / 紅線即時呼叫真人 + 交接卡落檔 → 全程具身(面向前方點頭/前傾 + 臉表情 + PIR 迎接/觸控安撫)**。這就是比賽 demo。

### 🟢 P2 — 打磨穩定性(demo 不炸)
雲端 429 自癒(#4)、開機自檢金鑰/權限(#7)、安全關鍵字熱重載(#8)、Android 生命週期(#9)。多數免實機已綠,真機只要確認不炸。
