## 一、為什麼需要實機(三類不確定性)

| 類別 | 不確定的事 | 主控台為何測不到 |
|---|---|---|
| **A. 硬體真值** | NeckZ 真實 ±40、頭/臂馬達動作幅度與時序、DOA 解析度、臉螢幕表情(Air S 無底盤) | Sim 用文字模擬,角度/時間都是假的 |
| **B. SDK 授權牆** | 搶麥、底盤 turn、LED、原生臉、觸控回呼,是否被 active-code 授權牆 **silent fail**(不報錯、就是沒反應) | 只有真 NuwaSDK aar 在真機上跑才知道 |
| **C. 雲端額度/延遲** | Robotics-ER 免費額度(已知 4 秒就 429)、視覺延遲 2.4–4.8s、Live session 時長上限/掉線 | 額度/延遲是 Google 端即時狀態 |

**關鍵洞察**:B 類(授權牆 silent fail)是**最該先驗的**——因為它會「無聲地廢掉一整類功能」。一個 5 分鐘的探針測試,就能決定後面幾天的方向。

## 三、我的建議優先序(待討論)

### 🔴 P0 — 先驗「前提」(便宜、會無聲廢掉整類功能的不確定性)
**目標:用最少時間,確認 SDK 授權牆放不放行。一旦某項 silent fail,後面的整合都要改路線。**
1. **① 搶麥** — 一個 30 秒測試:`stopListen()` → 錄 → STT 出文字?→ 決定所有語音功能可不可行。
2. **② 馬達(NeckZ/臂)+ ③ 觸控 + ⑨ 臉螢幕** — 命令 NeckZ→±40、抬臂、摸頭、播一個臉表情 → 一次驗完所有「免授權 `setMotor` / 觸控事件 / async 臉命令」授權牆放不放行。決定所有具身 + 觸控 + 表情功能。(Air S 無底盤,不必測 turn)

> P0 合計約 15–20 分鐘,**ROI 最高**:它把「未知」變「已知」,避免在會 silent fail 的路上投入幾天。Air S 不能移動,所以「頭 + 臂 + 臉螢幕 + 觸控」就是它的全部表達手段——這幾項能不能用,直接決定具身/表情功能的天花板。

### 🟡 P1 — 旗艦功能端到端(P0 通過後)
4. **⑧ Counselor 端到端 + ④ 具身共情幅度** — 專案最高價值場景;需要 P0①(搶麥)+②(轉向)。實際跑一段 🟢→🟡→🔴,順具身動作幅度。
5. **⑥ RoboGuide 指物**(相機光軸校 + 定點認物) — 需要 P0② + 開帳單;最有「只有機器人做得到」說服力的視覺 demo。

### 🟢 P2 — 需額外資源 / 長時
6. **⑥ RoboVisionTalk / SafetyEye**(看著物件對話 / 視覺安全之眼) — 需開帳單 + 相機;隱私之眼還要現場告知同意。
7. **⑦ Live 無縫重連** — 較長時間穩定性測試,排最後。

> ~~跨機 UDP / 雙機對話~~:雙機凱比對話屬實驗性質,不列入(口譯橋 #11 程式留著、CI 守著,不推進實機)。

### ⚙️ 決策先行(非實測,但 gating 上線)
- **Google Cloud 開帳單**:解 ER 額度 + ephemeral token(#16)。**gating ⑥⑦**。
- **安全關鍵字內容定稿**(輔導專業):#8 機制已備,內容 gating Counselor 上線。
- **會談記錄隱私規範**(存哪/誰讀/留多久,#5)+ **視覺用於輔導的現場告知同意**(#18):gating ⑧⑦。

## 四、建議的「第一次實機 session」腳本(約 30 分鐘,只做 P0)

> 一台真凱比、接 adb,照這個跑,就能決定後面方向:

1. **搶麥①**(~10 min):裝 Counselor APK → 無聲模式打字確認 LLM 通 → 切有聲,講一句 → 看 logcat STT 有沒有出文字。出 = 搶麥 OK;不出 = 試 `stopListen()` 前置或排除權限框暫停。
2. **馬達②+臉⑨**(~10 min):命令 NeckZ +40/−40、抬臂(肩 Y)、播一個臉表情 → 目視角度對不對、臉有沒有變 → 記校準值。(Air S 無底盤,不測 turn)
3. **觸控③**(~5 min):先跑參考專案 `~/Projects/UnityKebbi` 的 `touching.sh` 確認事件可達 → 再裝觸控 APK 摸頭看 logcat 有沒有 `onTouchEvent`。
4. **結論**(~5 min):三項各記 ✅/❌/需 workaround → 更新「必測①③⑤」狀態 → 決定 P1 路線。
