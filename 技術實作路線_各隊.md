# 各隊技術實作路線（依實測 Kebbi SDK｜對抗式審查後定稿）

> 硬體前提:競賽用會移動的輪式 Kebbi、共 3+ 台;開發機 H201 不會動(移動/多機需在輪式機與工作坊現場測)。
> 每條路線只建立在已驗證能力上,用 workaround 繞 active-code 授權牆,未驗能力一律附「真機實測 + 單機降級備案」。

### G1 小五(1人)｜技術實作路線

主攻:雙機接力闖關「教室地圖程式設計遊戲」— 2 台輪式 Kebbi 在地面接力走位 + Blockly 程式 + 交棒握手。
鐵律:全部建立在「已驗證可用」能力上;移動只能開迴路(速度×時間),無精準導航;多機與移動是否被授權牆擋皆未驗,必附實測+單機降級備案。

---

#### 1. 系統架構(資料流:感測 → 判斷 → 動作/語音回饋)

整體是一個「Blockly 指令佇列 → 開迴路執行 → 抵達定點交棒」的迴圈,兩台機分 A(先走)、B(接棒)。

```
[學生在 Blockly 拼程式]
        │  產生指令序列(前進N格 / 轉向 / 說話 / 交棒)
        ▼
[判斷層:Unity C# 指令直譯器]  ← 純 Unity 邏輯,不依賴任何 async SDK
        │
        ├─ 走位(移動)→ move(m/s) / turn(deg/s) × 計時器(開迴路) ⚠️移動是否被擋=必測(僅能在輪式機測)
        │                 地面膠帶標記 + 排練校時(每格固定秒數)
        │
        ├─ 轉向回饋 → 讀 getDirectionOfDOA()(免授權同步)→ 自寫 setMotorPositionInDegree 轉 NECK_Z
        │                 (取代被擋的 turnToDOA;DOA 角度須先夾限到 NECK_Z 物理可達範圍)
        │
        ├─ 語音回饋 → 雲端 TTS(UnityWebRequest HTTPS)→ Unity AudioSource 播放(取代內建 TTS)
        │
        ├─ 動作回饋(揮手/點頭/伸手交棒)→ 逐幀 setMotorPositionInDegree(取代 motionPlay)
        │
        └─ 交棒握手 → A 抵達定點 → 送訊息給 B(先試純 socket UDP/TCP)⚠️雙機端到端收送=必測(從未實跑)
                       B 收到 → 接力繼續走;失敗則退單機接力(見§7)
```

判斷層全部跑在 Unity C#,只呼叫「已驗證同步讀取 + 事件 + UnityWebRequest + setMotor」,把所有被擋的 async(turnToDOA/TTS/motionPlay)都換成自寫 workaround。移動與多機是僅有的兩個未驗環節,皆掛實測旗標,且兩者皆無法在 H201 桌上型機驗證(見§6)。

---

#### 2. 關鍵 SDK 呼叫

| 呼叫 | 類型 | 用途 | 授權狀態 |
|---|---|---|---|
| `setMotorPositionInDegree` | 免授權 同步 | 逐幀轉頭(NECK_Z)、揮手、伸手交棒動作 | 實測可用 |
| `getMotorPresentPossitionInDegree` | 免授權 同步 | 讀目前關節角,做動作回授/校正 | 實測可用 |
| `getDirectionOfDOA():float` | 免授權 同步 | 讀聲源角度,決定轉頭目標角(解析度/正後方/非語音=未驗) | 免授權 |
| `onWakeup(..., float direction)` / `onFaceSpeaker(float)` | 免授權 事件 | 喚醒/面向說話者觸發,onWakeup 已 H201 實測 | 免授權(onWakeup 已驗) |
| `UnityWebRequest`(任意 HTTPS) | 免授權 網路 | 雲端 TTS、(若需)雲端語音;無沙箱白名單 | 免授權,UDP 送已驗 |
| `move(m/s)` / `turn(deg/s)` | 移動命令 | 開迴路走位(max ~0.2m/s、~30deg/s) | ⚠️是否被授權牆擋=未驗,必測(僅輪式機可測) |
| 純 socket UDP/TCP(自寫) | 網路 | 雙機交棒訊息;UDP 送已驗、TCP 自寫收為全新未驗路徑、雙機端到端從未實跑 | 收未驗,必測 |

免授權同步/事件可直接信賴的:`setMotorPositionInDegree`、`getMotorPresentPossitionInDegree`、`getDirectionOfDOA`、`onWakeup`/`onFaceSpeaker`、`UnityWebRequest`。

---

#### 3. 繞授權牆 workaround(必採)

被擋的 async 命令在無 Active-code 授權時會「靜默 no-op」,絕不依賴。一律換成:

- **轉向**:不用 `turnToDOA`。改成讀 `getDirectionOfDOA()` 取得角度 → 自己用 `setMotorPositionInDegree` 把 `NECK_Z` 轉到該角度(必要時讀 `getMotorPresentPossitionInDegree` 做角度差校正)。
  - ⚠️**角度夾限(必做)**:DOA 是 0–360° 連續值,但 `NECK_Z` 物理轉動範圍有限(頭部不可能真的轉到正後方 180°),且 DOA 正後方/360° 範圍本身未驗。實作必須先把 DOA 目標角**夾限(clamp)到 NECK_Z 實測可達範圍**(用 `getMotorPresentPossitionInDegree` 與設角往復量出上下限),不得隱含「頭可轉到任意 360° 角度」。超出可達範圍時的處理:轉到最接近的可達端點(夾限),或退§7「依 Blockly 指令固定角度轉頭」。
- **說話**:不用內建 TTS。改成 `UnityWebRequest` 打雲端 TTS(可用中文即可,本隊非印尼語隊)取回音檔 → 載入 Unity `AudioSource` 播放。
- **動作(揮手/點頭/伸手交棒)**:不用 `motionPlay`。改成逐幀 `setMotorPositionInDegree` 對雙臂 8 馬達 + 頭 2 馬達做關鍵影格內插。
- **LED**:被擋,直接不依賴;改用 Unity 螢幕 UI(臉部表情/箭頭)做視覺回饋。

---

#### 4. 移動(本隊核心,僅開迴路)

- 只有連續速度命令 `move(m/s)`(max ~0.2)與 `turn(deg/s)`(max ~30);**無指定距離/角度/座標、無里程計、無避障、無 SLAM goto**。
- 走位做法:**速度 × 時間 開迴路** + **地面膠帶標記每一格/轉角** + **現場排練校時**(實測一格走幾秒、轉 90° 轉幾秒,寫成常數表)。Blockly 的「前進1格」就映射成「move(0.2) 持續 T 秒後 stop」。
- **明確承認無精準自動導航**:位置會累積誤差,靠膠帶定點與每關起點重新對位來吸收漂移;不得假設座標精準。
- **測試環境限制**:H201 是桌上型不會動,移動只能在「輪式會動的 Kebbi」與工作坊現場測;校時數據必須在比賽同款地面、同款機體上量。

---

#### 5. 多機(交棒握手)

- 方案優先序:**先試純 socket(自寫 UDP/TCP)** 在兩台 Kebbi 間送交棒訊息(UDP 送已驗);若需可靠收則用 TCP 自寫。
- ⚠️**風險明示**:純 socket **UDP 送已驗**,但 **TCP 自寫收為全新未驗路徑**、且 **UDP/TCP 雙機端到端收送從未實跑**。非公開 `ConnectionManager`(WiFi WebSocket P2P)Unity glue 不完整(缺 `startListen`/`send`),收未驗、從未雙機實跑 → 不作為第一方案;若要用須補 Java glue 並完成雙機收送實測。
- **前置必做(開工門檻)**:把「雙機端到端收送實測」列為多機任何功能的開工門檻 — 須**在工作坊現場/同網段、從 3 台以上實體機中取 2 台輪式機**跑一個最小程式:A 送一個字串、B 印出收到,確認端到端通(UDP 與 TCP 各驗一次)。沒過此測就走單機降級(§7)。
- 交棒協定極簡:A 抵定點 → 送 `HANDOFF`(含關卡序號)→ B 收到回 `ACK` → A 停、B 起步。用最短訊息降低收送失敗面。

#### 7. 單機/降級備案

- **移動被擋**:退成「定點演示版」— 機器人不移動,改用螢幕地圖 + 逐幀 setMotor 比手勢/轉頭表演「假裝走位」,Blockly 仍照常驅動指令直譯器,demo 程式設計概念完整。
- **雙機收送失敗(含 TCP 收不通)**:退成「單機接力版」— 同一台機跑完 A 段後在原地「換裝/換色 UI」象徵交棒,繼續跑 B 段;敘事仍是接力闖關。
- **DOA 不準/正後方無效/超出 NECK_Z 可達範圍**:轉頭回饋退成「依 Blockly 指令固定角度轉頭」(固定到 NECK_Z 可達端點),不依賴聲源即時值,亦不假設頭可轉任意角度。
- **雲端 TTS 斷網**:退成預先錄好的本地音檔用 AudioSource 播放(離線可放)。
- 任一未驗能力全失敗時,最低可 demo 版 = 單機 + 開迴路改定點表演 + 本地音檔 + Blockly 直譯器,仍能完整呈現「程式設計遊戲」主題,符合決賽完整性。

#### 8. 學生分工與難度(小五 1 人,依程式深度分模組)

單人專案,按難度分階段推進(由淺到深,先做能 demo 的核心):

1. **入門 / 可獨力**:Blockly 積木設計 + 指令直譯器(把積木轉成指令序列)。純邏輯,最貼近小五程度,也是 demo 主體。
2. **入門 / 可獨力**:逐幀 setMotor 揮手、點頭、伸手交棒動作(關鍵影格表),視覺好看、無授權風險。
3. **中階 / 需師輔**:開迴路走位校時(量秒數、貼膠帶、寫常數表),概念簡單但要耐心反覆量測(須在輪式機 + 比賽同款地面)。
4. **進階 / 師生共做**:DOA 讀值轉頭(getDirectionOfDOA → 夾限 → setMotor NECK_Z)。

### G2 國二｜技術實作路線

**主攻主題:幾何證明接力站 — 兩機帶讀證明每一步**
核心機制:甲機走到地面大圖的邊/角，用手臂指認 + 乙機宣告理由 + 雙機接力。

---

#### 1. 系統架構(資料流:感測 → 判斷 → 動作/語音回饋)

整體是「主控腳本驅動的回合制狀態機」,而非即時感測導航。每一步證明 = 一個 step state,內含「甲機要走到的標記點 + 要指的邊/角 + 乙機要唸的理由文字」。

```
[證明腳本 JSON]
   證明步驟序列(由學生事先編寫,每步=標記點代號 + 指向角度 + 理由台詞)
        │
        ▼
[主控狀態機(甲機 App 內)] ── 推進到第 N 步
        │
        ├─(甲機)走位:速度×時間開迴路 move/turn → 停在地面膠帶標記點
        │        ※無精準導航,靠排練校時 + 地面標記(見第4節)
        │
        ├─(甲機)指認:setMotorPositionInDegree 把手臂馬達轉到「指向該邊/角」的預設姿態
        │        ※逐幀 setMotor 取代被擋的 motionPlay(見第3節)
        │
        ├─(甲機 → 乙機)接力訊號:UDP/TCP socket 送「第 N 步完成,換你」
        │        ※純 socket 送已驗;收未驗,須先實測(見第5節)
        │
        ├─(乙機)宣告理由:雲端 TTS(Azure/Google,UnityWebRequest 取音檔)
        │        → Unity AudioSource 播放(取代被擋的內建 TTS)
        │
        └─(乙機)點頭/轉向觀眾:讀 getDirectionOfDOA() 或固定角 → setMotor 轉 NECK_Z
                 ※自寫轉頭取代被擋的 turnToDOA
                 ※僅限 NECK_Z/NECK_Y 機械可達範圍內;超出範圍不轉身(見第3、6節)
        │
        ▼
[接力回 step N+1] ← 乙機唸完送 socket 回甲機,推進狀態機
```

感測輸入(可選增強):觀眾出聲提問時用 `onWakeup(...,float direction)` / `getDirectionOfDOA()` 取得聲源角度,讓機器人「在頭部馬達可達範圍內轉頭面向提問者」再回答 — 此為加分互動,非主線必需。主線完全靠「事先編好的證明腳本 + 排練校時」推進,不依賴任何未驗的即時感測。

---

#### 2. 關鍵 SDK 呼叫

| 呼叫 | 用途 | 授權狀態 |
|---|---|---|
| `setMotorPositionInDegree` | 手臂指向邊/角、頭部轉向(NECK_Z/NECK_Y)、點頭動作(10 馬達:頭2+雙臂8) | ✅ 免授權同步,實測可用 |
| `getMotorPresentPossitionInDegree` | 讀目前馬達角,做動作校正/插值起點,並用於測頭部可達角度範圍 | ✅ 免授權同步,實測可用 |
| `getDirectionOfDOA():float` | 讀聲源角度(觀眾提問轉頭用) | ✅ 免授權同步 |
| `onWakeup(..., float direction)` / `onFaceSpeaker(float)` | 喚醒/說話者方向事件 | ✅ 免授權事件,onWakeup 已 H201 實測 |
| `UnityWebRequest`(任意 HTTPS) | 雲端 TTS 取音檔、(可選)LLM 生成理由說明 | ✅ 免授權,無沙箱白名單 |
| Unity `AudioSource` | 播放雲端 TTS 回來的音檔 | ✅ Unity 原生 |
| UDP/TCP socket(自寫) | 雙機接力訊號(送已驗、收未驗) | ✅ 送已實測 / ⚠️收需測 |
| `move(m/s)` / `turn(deg/s)` | 甲機開迴路走位(僅連續速度) | ⚠️是否被授權牆擋=未驗,列必測 |

被擋的 async(❌ 不可依賴,全部走 workaround):`turnToDOA`、內建 TTS、`motionPlay`、LED。

---

#### 3. 繞授權牆 workaround(必採)

- **轉向(面向觀眾/提問者)**:不呼叫 `turnToDOA`。改為 `getDirectionOfDOA()` 讀角度 → 自己用 `setMotorPositionInDegree` 把 NECK_Z 馬達轉到該角度。
  - ⚠️**重要限制(不得假設可面向任意角度)**:頭部 NECK_Z(左右)與 NECK_Y(上下)馬達有**機械轉動範圍上限**,無法只靠轉頭面向正後方或大角度聲源。NECK_Z/NECK_Y 的實際可達角度範圍**必須先實測**(見第6節新增列),確認自寫轉頭能覆蓋哪些 DOA 角度。
  - 對**落在頭部可達範圍內**的聲源:讀 DOA → setMotor 轉 NECK_Z 面向。
  - 對**超出頭部可達範圍**的聲源(如正後方、左右大角度):**不轉身**,改走第7節備案「固定面向觀眾」;或標註此情境需用 `move/turn` 轉動整台機身(而 `move/turn` 是否被授權牆擋本身已列第6節必測,未過則無此選項)。**絕不假設可靠轉頭面向任意 DOA 角度。**
  - 點頭(NECK_Y)同理:僅在 NECK_Y 實測可達範圍內做動作。
- **說話(宣告理由)**:不呼叫內建 TTS。改為 `UnityWebRequest` 打 Azure/Google TTS 取音檔 → Unity `AudioSource` 播放。台詞為中文,內建 TTS 雖支援中文但屬被擋 async,故一律走雲端確保不被 no-op。
- **動作(手臂指認、起手式)**:不呼叫 `motionPlay`。改為逐幀 `setMotorPositionInDegree` 插值控制雙臂 8 馬達,做出「指向邊/角」的姿態;可先用 `getMotorPresentPossitionInDegree` 取起點再線性內插,避免動作跳動。
- **LED**:不依賴。若要視覺提示,改用機身螢幕 UI 或 AudioSource 提示音。

---

#### 4. 移動(走位)

- **方案**:開迴路「速度 × 時間」。`move(max ~0.2 m/s)` / `turn(max ~30 deg/s)`,❌無指定距離/角度/座標、無里程計、無避障、無 SLAM goto。
- **走位實作**:地面貼膠帶標記每個證明步驟的停靠點與朝向 → 每段移動寫死「以 X m/s 跑 T 秒、以 Y deg/s 轉 T 秒」→ 現場排練反覆校時(電量、地面摩擦會影響,須當天場地校正)。
- **明確承認**:這不是精準自動導航,會有累積誤差;每步間插入「回到已知膠帶點」嘗試壓抑誤差。
- **⚠️回位本身也是開迴路、亦有誤差(不得暗示可靠精準回位)**:此「回到已知膠帶點」動作**同樣是開迴路速度×時間**,沒有里程計/座標回授,**回位動作自身也帶誤差**,只是用「靠視覺對齊已知膠帶標記 + 現場排練校時」來抑制誤差累積,**不是可靠的精準回位**。回位段的速度與時間參數**必須與一般移動段一起納入現場排練校時**,並接受其殘餘誤差。
- **測試限制**:移動只能在「輪式會動的 Kebbi」與工作坊現場測;H201 桌上型不會移動,無法在開發機驗走位。
- **⚠️未驗**:`move/turn` 是否也被授權牆擋未驗 → 列為必測(見第6節);若被擋,走第7節單機降級備案(定點機)。

---

#### 5. 多機(雙機接力)

- **通訊方案(優先序)**:
  1. **先試純自寫 socket(UDP 或 TCP)**:在兩機同一 WiFi 下,自寫 client/server 收送「接力訊號」(第 N 步完成 / 換手 / 同步開始)。純 socket「送」已實測;「收」未驗。
  2. **ConnectionManager(非公開 P2P WebSocket)為備選**:Unity glue 不完整(缺 startListen/send),需補 Java glue 才能用,風險高,不作為主線。
- **前置實測(必做,擋路項)**:**雙機收送實測** — 兩台實體機跑自寫 socket,確認「A 送 → B 確實收到並觸發回呼」,雙向都通。從未雙機實跑過,此項未過就不能上多機路線。
- **設計防呆**:接力訊號設計為「冪等 + 可重送 + 含 step 序號」,並加超時重送;一旦掉包,接收端靠序號去重,不會錯步。

#### 6. 必做真機實測(上線前要先驗的未知)

| 未知能力 | 怎麼測 |
|---|---|
| **頭部 NECK_Z(及 NECK_Y)馬達可達角度範圍** | 在 H201/輪式機,以 `getMotorPresentPossitionInDegree` 讀目前角,逐步 `setMotorPositionInDegree` 推到左右/上下極限,測出 NECK_Z(左右)與 NECK_Y(上下)實際機械可達角度範圍;對照可覆蓋哪些 DOA 角度。超出範圍的聲源(如正後方)明確標註:**不轉身、改走備案固定面向觀眾**,或需 `move/turn` 轉身(而 move/turn 是否被擋本身已列本表必測);**不得假設可轉頭面向任意角度**。此項可在 H201 桌上型先測(頭部馬達不需移動)。 |

| **走位開迴路精度與校時(含回位段)** | 在輪式機+地面膠帶,實跑每段「速度×時間」與「回到已知膠帶點」回位段,量測停點誤差,記錄當天場地校正參數;回位段與一般移動段同為開迴路、同須校時。 |

| (可選)**相機/搶麥** | 本主題主線不需相機/PCM 搶麥;若做觀眾提問 STT 才需測 WebCamTexture 開相機權限與 stopListen 釋放後搶麥,否則略過。 |

#### 7. 單機/降級備案(任一未驗能力失敗仍能 demo)

- **多機收送失敗** → 退成**單機分飾兩角**:同一台機用兩種語音音色(雲端 TTS 切 voice)輪流扮演「指認者」與「理由宣告者」,接力靠機內狀態機推進,完全不用 socket。完整證明流程不受影響。
- **移動被授權牆擋(或開迴路太不穩)** → 退成**定點機**:機器人不移動,放在地面大圖固定位,改用「手臂指向 + 雷射筆/長指棒道具 + 螢幕高亮對應邊角圖」完成指認;走位由人工或腳本省略,證明步驟照唸。
- **頭部馬達可達範圍不足以面向某聲源(或 DOA 不可靠)** → 觀眾互動轉頭功能在超出範圍/不可靠時關閉,改**固定面向觀眾**;主線證明不依賴 DOA 與轉頭,不影響完整性。對範圍內聲源仍可做小角度轉頭增強。
- **雲端 TTS 失敗(斷網)** → **預先把所有證明台詞離線快取成音檔**打包進 App,AudioSource 直接播本地檔,無網也能跑完整 demo。

決賽完整性保證:最差情況=「單機 + 定點 + 固定面向觀眾 + 本地快取音檔 + 手臂指認」,仍可完整走完整套幾何證明帶讀,無任何被擋 async、無精準導航假設、無「可面向任意角度」假設。

### G3 國三升高一｜技術實作路線

主攻:Kebbi 體育課:動作鏡像教練
核心:逐幀關節示範 + DOA 轉頭面向發問學生 + 攝影機評學生動作

---

#### 1. 系統架構(資料流:感測 → 判斷 → 動作/語音回饋)

```
[感測層]
 ├─ DOA 聲源角度
 │    ・輪詢 getDirectionOfDOA():float                          ← 免授權 sync(連續度數,已驗讀取)= 主轉頭角度來源(安全主幹)
 │    ・事件 onWakeup(...,float direction)                       ← 免授權事件(H201 已實測)= 系統喚醒模式下的觸發+角度
 │    ・事件 onFaceSpeaker(float)                                ← 免授權事件,但「是否實際觸發」未驗 → 次要來源,通過第6節實測前不得作為唯一角度來源
 ├─ 相機影像(未驗)
 │    ・Unity WebCamTexture 自取畫面 → 自寫 MediaPipe 跑骨架/臉框  ← Nuwa 不給像素,須自取
 └─ 麥克風(未驗,單一擁有者,兩種模式互斥,見下)
      ・模式A 系統喚醒詞偵測:系統持麥 → 觸發 onWakeup(免授權事件)
      ・模式B 自由語音:stopListen() 釋放系統麥 → Android AudioRecord 自取 PCM → 雲端 STT
      ・⚠️同一時間只能一個持麥;進入模式B 後 onWakeup 可能失效,離開模式B 須還麥回模式A

[判斷層](全部在 Unity/C# 本地或雲端)
 ├─ DOA 角度 → 換算成 NECK_Z 目標角(本地計算;角度以 getDirectionOfDOA 為準)
 ├─ MediaPipe 骨架關鍵點 → 與「標準動作模板」比對 → 評分/糾錯文字(本地計算)
 └─ STT 文字 → LLM(雲端 HTTPS)→ 回應文字 → 雲端 TTS

[動作/語音回饋層]
 ├─ 轉頭面向學生:讀 getDirectionOfDOA → 自己 setMotorPositionInDegree(NECK_Z, 角度)  ← workaround,不用 turnToDOA
 ├─ 逐幀關節示範:setMotorPositionInDegree(雙臂8馬達 + 頭2)逐幀播     ← workaround,不用 motionPlay
 └─ 語音講解/糾錯:雲端 TTS 音檔 → Unity AudioSource 播放            ← workaround,不用內建 TTS
```

說明:整條主線(感測 DOA、計算、馬達回饋)完全建立在「免授權同步/事件 + 自寫馬達」上,被擋的 async 命令一律以 workaround 取代。主轉頭角度以**已驗的 getDirectionOfDOA(sync 讀取)**為安全主幹;onWakeup(已實測事件)在系統喚醒模式下提供觸發與角度;onFaceSpeaker 僅為次要來源,須先通過「是否觸發」實測。相機與搶麥屬未驗,各自配實測 + 降級備案(見 6、7)。**麥克風單一擁有者導致「系統喚醒詞偵測(onWakeup)」與「AudioRecord 自由語音」為互斥模式**,架構上明訂同一時間只能一個持麥(見第 3、6、7 節)。

#### 3. 繞授權牆 workaround(鐵律:被擋 async 一律替換)

1. **轉頭面向發問學生**(替 `turnToDOA`):
   主來源 `getDirectionOfDOA()`(已驗 sync 讀取)取得角度 → 換算 NECK_Z 目標角 → `setMotorPositionInDegree(NECK_Z, 角度)`。轉動中用 `getMotorPresentPossitionInDegree` 做緩動/限位。
   ・系統喚醒模式下可用 `onWakeup(...,direction)`(已實測)做「被喚醒當下」的觸發與角度。
   ・`onFaceSpeaker(deg)` **僅作次要/輔助來源**,且必須先通過第 6 節「是否實際觸發」實測,未通過前**不得作為唯一角度來源**;主幹一律落在 `getDirectionOfDOA`。

2. **講解與糾錯語音**(替內建 TTS):
   回應文字 → `UnityWebRequest` 打雲端 TTS(Azure/Google)→ 拿音檔 → Unity `AudioSource.Play()`。印尼語同走此路(內建僅 7 語,無印尼語)。

3. **逐幀關節示範動作**(替 `motionPlay`):
   把標準體育動作切成關鍵幀,逐幀對頭2+雙臂8共 10 馬達呼叫 `setMotorPositionInDegree`,以固定 timestep 推進,做出「鏡像教練」示範。

4. **麥克風模式互斥(架構鐵律,非單純 workaround)**:
   麥克風為單一擁有者,故明訂兩個**互斥模式**,同一時間只能一個持麥:
   - **模式A 系統喚醒詞偵測**:系統持麥,可收 `onWakeup`;**此時不可開 AudioRecord**。
   - **模式B AudioRecord 自由語音**:先 `stopListen()` 釋放系統麥 → 開 `AudioRecord` 取 PCM 跑雲端 STT;**此時 onWakeup 可能失效**(系統不再持麥)。
   切換時機:對話一輪結束、或自由語音逾時無輸入 → 關閉 AudioRecord、還麥回系統,回到模式A 等待下一次喚醒。**絕不同時並列 onWakeup + AudioRecord 於同一時刻持麥**(避免 C 組同時依賴兩個互斥未驗行為)。兩者共存衝突須先由第 6 節「搶麥 vs 喚醒並存測試」釐清真機行為後,再定切換邏輯。

#### 6. 必做真機實測(上線前驗未知,每項一句怎麼測)

1. **相機權限**:在 Kebbi OS 上跑一支只開 `WebCamTexture` 的最小 app,看是否拿得到非黑畫面 frame。
2. **搶麥**:在系統持麥(待喚醒)狀態下呼叫 `stopListen()` 再開 `AudioRecord`,確認能否取得 PCM 且系統不報錯。

4. **移動是否被擋**:在輪式機下 `move(0.1)` 1 秒,看車體是否真的動(動=未被擋;靜默不動=被授權牆擋)。
5. **多機收送**:兩台機一送一收跑自寫 socket,確認封包雙向到達。
6. **DOA 解析度/盲區 + onFaceSpeaker 是否觸發**:固定距離繞 Kebbi 一圈每 30° 出聲,記錄 `getDirectionOfDOA()` 回報值,確認解析度、正後方、非語音聲是否更新;**同時先確認 `onFaceSpeaker` 是否實際 fire**——未觸發即整條主線只用 `getDirectionOfDOA`(+ onWakeup direction),不把 onFaceSpeaker 當角度來源。
7. **馬達逐幀流暢度**(已驗能力的工程化):量逐幀 `setMotor` 的可達幀率,定 timestep 避免抖動。

### G4 高一升高二｜技術實作路線

主題：Tebak Arah - 印尼語方位定向遊戲
核心：DOA 當「方位對錯」真值（**前置硬門檻：第 6.3 實測通過後才成立**）+ 頭轉面向學生 + 印尼語問答 + LLM 出題糾錯

---

#### 0. DOA「已驗 / 未驗」邊界（先界定，貫穿全文）

DOA 是本遊戲的「方位真值」來源，但其能力被切成兩半，必須嚴格區分：

- **已驗（可直接依賴）**：
  - `getDirectionOfDOA():float` 連續讀值本身可讀（免授權、同步）。
  - `onWakeup(..., float direction)` 喚醒事件帶聲源角（H201 已實測觸發）。
- **未驗（用到就必須附實測 + 降級，視同硬門檻）**：
  - 連續值 →（八）方位**量化準度**（誤差是否小到能當對錯真值）。
  - **正後方 / 360° 是否更新**（盲區風險）。
  - **非語音聲音是否觸發 / 更新**。
  - `onFaceSpeaker(float)` 的**解析度與觸發更新**。

> ⚠️ 規則：凡在第 1、3 節「把 DOA 量化方位當對錯真值」之處，皆**懸於第 6.3 前置實測**。此為硬門檻，類比第 5 節多機的「雙機收送實測必須先過」——**第 6.3 不過，方位真值降級走第 7 節，不得直接上線**。

#### 1. 系統架構（資料流：感測 → 判斷 → 動作/語音回饋）

```
[出題] LLM 雲端出題(UnityWebRequest→LLM)
        → 雲端印尼語 TTS(UnityWebRequest→Azure/Google)
        → AudioSource 播「請站到 Kebbi 的東邊」(Utara/Selatan/Timur/Barat)
            │   ※站位限制在「第 6.3 實測通過的可靠扇區」內(見第 3、6.3、7 節)
            │
[學生回應/發聲] 學生在某方位喊印尼語答案
            │
[感測] onWakeup(...,float direction)(已驗) 觸發開一輪
        + getDirectionOfDOA():float 讀連續聲源角度(讀值已驗)         ← 方位真值候選
            ⚠️「連續值→方位量化準度/正後方更新/非語音觸發」未驗 = 硬門檻(第 6.3)
        + (麥克風 PCM 自取 AudioRecord，需先 stopListen 釋放) → 雲端印尼語 STT 取得學生口說答案
        + (onFaceSpeaker 不作為任何判定/動作的必要輸入；僅選用，失敗可直接停用)
            │
[判斷] 在 Unity C# 內：
        - 把 DOA 角度量化成方位 → 與題目要求方位比對 = 對/錯真值
          ⚠️僅在第 6.3 通過、且只用「可靠扇區方位」時成立；否則走第 7 節降級
        - 把 STT 文字 + DOA 方位一起丟 LLM → 判定答案正確性 + 產生糾錯講評(印尼語)
            │
[動作回饋] 讀 DOA 角度(getDirectionOfDOA，已驗) → 自寫 setMotorPositionInDegree 把 NECK_Z 轉到該角(頭轉面向學生)
            + 雙臂 8 馬達逐幀 setMotor 做「指向/鼓掌/搖頭」表情動作
[語音回饋] LLM 講評文字 → 雲端 TTS → AudioSource 播印尼語回饋
```

資料流每一步都落在「已驗證可用」「明確 workaround」或「附前置實測+降級的未驗環節」上：DOA 連續讀值/onWakeup（已驗）、馬達（已驗）、UnityWebRequest（已驗 HTTPS）。**DOA 方位量化準度、正後方更新、搶麥、相機、移動是否被擋、多機**全部走第 6 節實測 + 第 7 節降級。

#### 3. 繞授權牆 workaround（鐵律：被擋 async 一律改寫）

- **轉向面向學生**：不呼叫被擋的 `turnToDOA`。改為 `getDirectionOfDOA()` 讀角 → 自己用 `setMotorPositionInDegree(NECK_Z, 角度)` 把頭轉到該方位；用 `getMotorPresentPossitionInDegree` 做分段插值平滑轉頭，避免甩頭。**頭轉僅依賴已驗的 `getDirectionOfDOA` 連續讀值；`onFaceSpeaker` 不作為頭轉/判定的必要輸入**（僅選用平滑來源，第 6.6 實測不過即停用，不影響頭轉）。
  - ⚠️ **方位量化用於對錯真值的前置硬門檻**：把連續角量化成方位、並以該方位判對錯，**只有在第 6.3 實測通過、且僅使用「可靠扇區方位」時才成立**（類比第 5 節多機前置門檻）。第 6.3 不過 → 依第 7 節降級（縮減方位 / 改相機臉框真值 / 改人工確認），**不得直接以未驗的八方位精度上線**。
- **正後方盲區設計（場佈即規避）**：在膠帶標記階段，就把站位**限制在第 6.3 實測可靠的扇區內**；若正後方判定為「不更新」，規則中**明確聲明不使用正後方位**（例如只用前方半圈 + 兩側，避開 Kebbi 背面 90°~120° 區）。場佈與題目產生器都只允許可靠方位。
- **說話（印尼語）**：不呼叫被擋的內建 TTS（且內建僅 7 語、無印尼語）。改為 `UnityWebRequest` → Azure/Google 印尼語 TTS 取得音檔 → Unity `AudioSource.Play()` 播放。
- **動作（指向/鼓掌/搖頭）**：不呼叫被擋的 `motionPlay`。改為雙臂 8 馬達 + 頭 2 馬達逐幀 `setMotorPositionInDegree` 自編關鍵影格序列（在 Unity 用協程逐幀寫角）。
- **回饋燈光**：不依賴被擋的 `LED`；視覺回饋改用頭/臂動作 + Unity 端螢幕/UI（若有面板）呈現。

---

#### 4. 移動（本主題不需精準導航，移動為選用加分）

核心玩法靠「學生移動到方位 + DOA 判定」，Kebbi 本體可全程定點，**不依賴移動**。若要加分讓 Kebbi 微調朝向或小幅移動：

- 只有連續速度 `move(m/s, max~0.2)` / `turn(deg/s, max~30)`，**無指定距離/角度/座標、無里程計、無避障、無 SLAM goto**。
- 走位只能「**速度 × 時間 開迴路** + 地面膠帶標記停位 + 現場排練校時」，明確承認**無精準自動導航**，不得假設座標到位。
- `move/turn` 是否也被授權牆擋 = **未驗**，列入第 6.4 必測。
- 桌上型 H201 不會移動，移動相關**只能在輪式機與工作坊現場測**。

---

#### 5. 多機（選用：多台 Kebbi 對戰/接力出題）

- 預設**單機即可完整 demo**；多機為加分（共 3 台以上實體機可用）。
- 通訊方案順序：① 先試**純 socket 自寫**（UDP 送已驗、TCP/收未驗）在 Unity 端做 Kebbi↔Kebbi 訊息（誰出題、誰判定）；② 非公開 `ConnectionManager`（WiFi WebSocket P2P）Unity glue 不完整（缺 startListen/send），需**補 Java glue** 才能收送。
- **前置硬門檻**：「雙機收送實測」必須先過（從未雙機實跑）。未過 → 退單機。

---

#### 6. 必做真機實測（上線前先驗的未知，每項配測法 + 通過/退場準則）

1. **搶麥（6.1，第一必測）**：系統持麥（喚醒/內建語音）時，自寫 `AudioRecord` 先 `stopListen` 釋放後能否成功 open 麥克風取得 PCM？
   測法：寫最小 app，喚醒後立即 stopListen→AudioRecord.startRecording，檢查是否拿到非空 PCM 且無 exception。
   通過：連續取得非空 PCM、無 exception；退場：拿不到 PCM 或拋例外 → 第 7 節「搶麥失敗」降級。
2. **相機權限（6.2）**：Kebbi OS 是否准一般 app 開 `WebCamTexture`？
   測法：最小 app 嘗試開相機抓一張 frame，看是否回傳像素或被拒。
   通過：拿到非空像素 frame → 可作為「DOA 失敗時的臉框方位替代真值」；退場：被拒 → 不啟用 MediaPipe，DOA 失敗時走人工確認保底（第 7 節）。
3. **DOA 解析度 / 盲區 / 正後方 / 非語音（6.3，方位真值硬門檻）**：DOA 在各方位下的角度誤差、**正後方是否更新**、非語音聲音是否觸發？
   測法：在膠帶標好的方位各喊一句印尼語，記錄 `getDirectionOfDOA()` 回值與真值偏差；**正後方（Kebbi 背面）單獨測一組**，並另測一次拍手/敲擊（非語音）。
   - **量化準度準則**：八方位下平均誤差 ≤ 半格（≤ 22.5°）且無相鄰格誤判 → 八方位通過；否則**降為僅 4 方位**重測，4 方位下誤差 ≤ 45°/2 → 4 方位通過。仍不過 → 第 7 節「DOA 解析度失敗」降級。
   - **正後方準則（明確判定）**：對正後方連喊 3 次，`getDirectionOfDOA()` 回值需在合理誤差內**確實更新到背面角區**，3 次皆更新=「正後方可用」；**任一次不更新或卡在前方角=「正後方判定為不更新」→ 場佈與題庫一律排除正後方位**（只用可靠扇區，見第 3 節）。
   - **非語音準則**：拍手是否讓回值更新；不更新即規則限定學生需發語音（喊印尼語單字），不依賴非語音觸發。
4. **移動是否被授權牆擋（6.4）**：無授權時 `move/turn` 是否靜默 no-op？
   測法：輪式機下發 move(0.1) 1 秒，觀察是否實際位移。通過=有位移；退場=不動 → 第 7 節「移動被擋」降級（全程定點）。
5. **多機收送（6.5）**：純 socket / ConnectionManager 雙機能否互收訊息？
   測法：A 機送固定字串，B 機印出收到內容，雙向各一次。通過=雙向都收到；退場=收不到 → 退單機。
6. **onFaceSpeaker 解析度/觸發（6.6，選用環節）**：`onFaceSpeaker(float)` 是否觸發、回值與真值偏差？
   測法：在 4 個方位各發聲一次，記錄事件是否觸發與回值偏差。通過=穩定觸發且偏差可用 → 可作頭轉平滑輔助；退場=不觸發/偏差大 → **直接停用 onFaceSpeaker**，頭轉純靠已驗的 `getDirectionOfDOA`（不影響任何判定/動作）。

#### 7. 單機/降級備案（任一未驗失敗仍能 demo）

- **搶麥失敗**（拿不到 PCM/搶不到麥）：放棄學生口說印尼語答案；改為「**DOA 方位即答案**」——學生只要站到正確方位喊任何語音，用 `getDirectionOfDOA()` 判方位對錯（前提：第 6.3 通過）；題目與糾錯仍由 LLM 出、雲端 TTS 播。
- **DOA 解析度/盲區失敗（新增，對應第 6.3）**：當第 6.3 顯示八方位量化誤差過大或正後方不更新時，依序降級：
  - (a) **縮減方位**：只用前方/側方可靠的少數方位（例如僅 4 方位且避開正後方），場佈膠帶與 LLM 題庫只允許這些方位，仍由 DOA 判對錯。
  - (b) **改相機真值**：若 4 方位仍不可靠，改用第 6.2 通過的 `WebCamTexture` + MediaPipe **臉框中心推方位**當替代真值（臉框中心方向 → 量化方位）；DOA 退為僅做頭轉朝向（不當判定真值）。
  - (c) **人工確認保底**：若相機亦被拒（第 6.2 退場），改為「**LLM 出題 + 學生站位由教師/螢幕人工確認**」——螢幕顯示題目要求方位，教師或裁判按鈕確認學生是否站對，LLM 給糾錯講評。**完全不依賴未驗的 DOA 八方位精度**。
- **相機失敗**（WebCamTexture 被拒）：若僅作備援則本就不啟用 MediaPipe；若是被 DOA 失敗叫來當替代真值卻也失敗，直接落到上面 (c) 人工確認保底。
- **移動被擋**：Kebbi 全程定點，只靠 NECK_Z 轉頭面向學生（轉頭走 setMotor，不受移動牆影響）。
- **onFaceSpeaker 失敗**：直接停用，頭轉純靠 `getDirectionOfDOA`（已驗），不影響任何判定/動作。
- **多機失敗**：退**單機**，一台 Kebbi 主持整場問答，仍是完整決賽 demo。
- **雲端不穩**（LLM/STT/TTS 連線失敗）：預先本地快取一組印尼語題庫 + 預錄 TTS 音檔，離線可跑固定題流程。

**底線版本（全部未驗能力都失敗時，重寫——不再單靠未驗的 DOA 八方位精度）**：
單機定點 + 本地印尼語題庫 + 預錄印尼語音檔（雲端不可用）+ setMotor 轉頭/動作回饋 +
**站位由「LLM/題庫出題 → 螢幕顯示要求方位 → 教師/裁判人工確認對錯」**（DOA 即使失敗也不影響）。
若第 6.3 通過則可把人工確認升級為 DOA 自動判定；若第 6.3 只過 4 方位則自動判定限可靠扇區。
→ 無論 DOA / 搶麥 / 相機 / 移動 / 多機 / 雲端任意組合失敗，皆有可完整演出的遊戲。

### G5 高三升大學｜技術實作路線

主攻主題:歷史法庭辯論劇場。核心機制＝兩機分飾正/反兩方、中央移動逼近對峙 + DOA 轉向發言學生 + 手臂手勢 + LLM 生成辯詞。本路線只建立在「已驗證可用」能力上;凡被授權牆擋的 async 一律改 workaround;凡未驗能力(移動是否被擋/多機收送/DOA 解析度/搶麥/相機)一律附「先實測 + 單機降級備案」。**不假設任何精準導航。**

---

#### 1. 系統架構(資料流:感測 → 判斷 → 動作/語音回饋)

```
[發言學生講話]
   │
   ▼
(感測) 主路徑唯一可靠來源:
        getDirectionOfDOA():float   ← 免授權同步輪詢(已是免授權可用),連續度數,主路徑只信這個
        輔助(加分,未驗,不放主路徑):
        onWakeup(...,float direction) 事件  ← 免授權,H201 已實測(僅喚醒當下拿一次方位)
        onFaceSpeaker(float) 事件            ← 免授權,但「是否持續觸發+解析度」皆未驗 → 僅加分
   │  得到聲源角度 θ
   ▼
(判斷-本機) 機A/機B 各自的「辯論狀態機」(Unity C#):
        - 現在輪到哪一方發言?(回合制,由主控機決定)
        - 把 θ 換算成 NECK_Z 目標角度
        - 組 prompt:歷史主題 + 立場(正/反) + 對方上一句
   │
   ├─(動作) 轉頭看學生:輪詢 getDirectionOfDOA() → setMotorPositionInDegree(NECK_Z, θ)  ← 取代被擋的 turnToDOA
   │        手勢(指對方/舉手陳述):逐幀 setMotorPositionInDegree(雙臂8馬達)  ← 取代被擋的 motionPlay
   │
   ├─(語音) LLM 生成辯詞:UnityWebRequest → HTTPS(Claude/GPT API)  ← 免授權
   │        雲端 TTS:UnityWebRequest → Azure/Google TTS → 回傳音檔
   │        → Unity AudioSource.Play()  ← 取代被擋的內建 TTS
   │
   └─(多機協調) 機A↔機B 交換「換你發言/我這句講完了/對方論點摘要」:
            自寫 socket:UDP 送已實測;UDP 收、TCP 送/收皆未驗 → 列前置實測
            ConnectionManager 走 WiFi WebSocket P2P 但 Unity glue 不完整 → 列前置實測
```

要點:**所有「會被授權牆擋的 async」都不出現在主路徑上**,主路徑只用「免授權同步輪詢(getDirectionOfDOA)+ 事件(onWakeup)+ 自寫馬達 + 雲端 HTTP + AudioSource」。**DOA 主路徑以 getDirectionOfDOA():float 輪詢為唯一可靠來源;onWakeup 只在喚醒當下用一次,onFaceSpeaker 僅加分、不依賴。**

---

#### 2. 關鍵 SDK 呼叫

| 呼叫 | 類別 | 用途 |
|---|---|---|
| `getDirectionOfDOA():float` | **免授權,同步輪詢(主路徑唯一可靠 DOA 來源)** | 取得聲源角度,驅動轉頭(每幀/定時輪詢) |
| `onWakeup(..., float direction)` | **免授權,事件**(H201 已實測) | 喚醒當下第一時間拿一次方位(非持續追蹤) |
| `onFaceSpeaker(float)` | **免授權,事件(是否持續觸發+解析度皆未驗)→ 僅加分** | 若實測確認會持續觸發,才當作 getDirectionOfDOA 之外的加分輔助 |
| `setMotorPositionInDegree(motor, deg)` | **免授權,實測可用** | 轉頭(NECK_Z)、手臂手勢(雙臂8馬達)、逐幀動作 |
| `getMotorPresentPossitionInDegree(motor)` | **免授權,實測可用** | 讀目前角度,做插值/安全限位/回中 |
| `UnityWebRequest`(任意 HTTPS) | **免授權,實測可用** | LLM 辯詞生成、雲端 STT/TTS、UDP 送(已測) |
| `AudioSource.Play()`(Unity 原生) | 非 SDK 授權牆 | 播雲端 TTS 回來的音檔 |
| `WebCamTexture`(Unity 原生) | 非 SDK 授權牆(但 OS 權限未驗) | 自取影像跑 MediaPipe(本隊非必要,列為加分) |

被擋(不直接用):`turnToDOA`、內建 TTS、`motionPlay`、LED。move/turn 速度命令是否被擋＝未驗,見第6節。

#### 3. 繞授權牆 workaround(鐵律)

- **轉向發言學生**:不呼叫 `turnToDOA`(async,可能靜默 no-op)。改:輪詢 `getDirectionOfDOA()` 取得 θ(主路徑唯一可靠來源;`onWakeup` 只在喚醒當下取一次初始方位)→ 自寫插值把 `NECK_Z` 用 `setMotorPositionInDegree` 轉到 θ(每幀讀 `getMotorPresentPossitionInDegree` 做平滑、限位、防超程)。`onFaceSpeaker` 僅在第6節實測確認「會持續觸發」後才作加分輔助,不放主路徑。
- **說話(中/印尼語辯詞)**:不呼叫內建 TTS。改:`UnityWebRequest` 打 Azure/Google TTS(含印尼語,內建僅7語不含印尼語)→ 取回音檔 → `AudioSource.Play()`。
- **手臂手勢(指對方、舉證、攤手)**:不呼叫 `motionPlay`。改:把每個手勢做成「關節角度關鍵幀序列」,以 `setMotorPositionInDegree` 逐幀播放(雙臂8馬達),頭部2馬達同步配合。
- **燈效**:LED 被擋 → 改用螢幕 UI 或 AudioSource 音效代替「換邊發言」的提示。

#### 4. 移動(中央逼近對峙)

- 只有連續速度命令:`move(m/s)`(max ~0.2 m/s)、`turn(deg/s)`(max ~30 deg/s)。**無指定距離/角度/座標、無里程計、無避障、無 SLAM goto。**
- 走位方式:**速度 × 時間 開迴路** + **地面膠帶標記起點/對峙點/退場點** + **現場排練校時**(實測每段「幾秒 × 速度」剛好停在膠帶)。明確承認:**這不是精準自動導航,會漂移,需現場每場校準。**
- 「逼近對峙」的腳本化:雙方各跑固定秒數朝中央 `move`,停 → 開始辯論回合;不靠感測判定到位,靠膠帶+排練。
- ⚠️ **只能在輪式機 / 工作坊現場測**(H201 桌上型不會移動)。且 **move/turn 是否也被授權牆擋＝未驗**(見第6節),若被擋則整段移動退為靜態擺位(見第7節)。

#### 5. 多機(機A正方 / 機B反方)

- 需求:雙機交換「回合控制 + 對方上一句論點摘要」,讓 LLM 接話。
- 通訊方案(依風險排序):
  1. **自寫純 socket**:**僅「UDP 送」已實測**;**UDP 收、TCP 連線/送/收皆未驗**。先用 UDP 送「換你發言」這類短訊;若需可靠順序而考慮 TCP,須注意 TCP 在此平台未驗,不能當作現成可靠選項,必須納入第6節前置實測一起驗。
  2. ConnectionManager(非公開,WiFi WebSocket P2P):Unity glue 不完整(缺 `startListen`/`send`),需**補 Java glue**;純 socket 送已驗、**收未驗、從未雙機實跑**。
- **前置硬性條件:雙機收送實測必須先過**(A 送、B 真的收到並觸發狀態切換,且 UDP 收 / TCP 送收 分別驗過),否則多機不上線。
- 協調設計刻意做薄:只傳「回合 token + 上一句摘要文字」,避免複雜同步;LLM 生成放在各自本機,降低跨機依賴。

### 合體彩蛋｜技術實作路線

> 主攻：**G2×G3×G5 多機協作大劇場（G5 中控）**。核心機制：3+ 台 Kebbi 由 G5 中控廣播 cue、各站接力演出、壓軸同步走位。
> **最高風險＝多機通訊（從未雙機實跑）+ 移動（速度命令是否被授權牆擋未驗）**。本路線只建立在「已驗證可用」能力上，所有被擋 async 一律換 workaround，所有未驗能力都附「先實測＋單機降級備案」，不假設任何精準自動導航。

> **語音範圍聲明（避免誤判遺漏 G4 搶麥實測）**：本路線語音**只做輸出**（雲端 TTS → AudioSource 播放），**不自取麥克風 PCM、不做 STT/語音辨識**，因此**不涉及「搶麥（系統持麥能否搶麥）」這個未驗點**，第 6 節必測清單故不需列入 G4 搶麥測試。喚醒方位只用 SDK 免授權事件 `onWakeup(...,float direction)`，不讀 PCM。

#### 1. 系統架構（資料流：感測 → 判斷 → 動作/語音回饋）

整體採「**中控腦（G5）+ 各站手腳（G2/G3/G5夥伴機）**」的星狀架構，cue 由中控單向廣播、各站 ack 回報。

```
[感測層]
  ├ DOA 聲源角度：getDirectionOfDOA():float / onWakeup(...,float direction)（免授權，同步讀+事件；不讀麥克風 PCM）
  └ （選配視覺）Unity WebCamTexture 自取影像 → 自寫 MediaPipe 算臉框中心 → 推方位（相機權限未驗）

        │
        ▼
[判斷層 = G5 中控]
  ├ 演出狀態機（scene → cue → next）：純 Unity C# 邏輯，不靠 SDK
  ├ LLM（選配主持詞生成）：UnityWebRequest 任意 HTTPS（免授權）
  └ 計時/計分/交棒握手：Unity 內部變數

        │  廣播 cue（{from, role, cue, state, ts}）
        ▼
[通訊層 = 多機] 先試自寫 socket（UDP 已驗送 / TCP），ConnectionManager 需補 Java glue
  G5 ──cue──▶ G2站機、G3站機、G5夥伴機；各站 ──ack──▶ G5

        │
        ▼
[動作/語音回饋層 = 各站機]
  ├ 轉頭/指向/逐幀動作：setMotorPositionInDegree（NECK_Z + 雙臂 8 軸）（免授權，實測可用）
  ├ 讀目前角度回授：getMotorPresentPossitionInDegree（免授權）
  ├ 說話（僅輸出）：雲端 TTS（Azure/Google）經 UnityWebRequest → Unity AudioSource 播放（繞內建 TTS；不做 STT）
  └ 走位（壓軸）：move(m/s)/turn(deg/s) 速度×時間開迴路 + 地面膠帶標記（移動是否被擋未驗）
```

**關鍵設計**：中控的「判斷層」全是純 C# 狀態機，不依賴任何被擋 async；它對外只用兩條已驗證管道——UnityWebRequest（雲端）與 socket（多機）。各站機收到 cue 後，所有實體輸出都走「免授權同步 setMotor + AudioSource」，把授權牆風險降到只剩「移動」與「多機」兩個未驗點。語音純輸出、不取 PCM，故不引入搶麥未驗點。

#### 4. 移動（壓軸同步走位）

- **方法**：只有連續速度 `move(m/s)`（max ~0.2）/`turn(deg/s)`（max ~30），**無指定距離/角度/座標、無里程計、無避障、無 SLAM goto**。因此走位採「**速度 × 時間開迴路**」：例如「以 0.15m/s 前進 4 秒 ≈ 0.6m」。
- **地面膠帶標記**：舞台貼 home 點與終點膠帶，每台走到標記停；現場排練反覆校時微調秒數，吸收電量/地面摩擦造成的誤差。
- **同步走位**：中控廣播 `cue: walk_to_center`，各站機**同時**起跑各自預排好的「速度×時間」腳本；以排練校時對齊節奏，不靠任何自動導航。
- **誠實聲明（呼應鐵律）**：❌ 無精準自動導航。**本路線不假設避障、不假設里程計**；終點停靠**靠膠帶標記 + 排練校時的開迴路秒數**達成，**並非任何感測器觸發停止**。安全間距由排練保證、現場每台一人手持緊急停止。
- **測試限制**：H201 是桌上型不會移動 → **移動只能在輪式機與工作坊現場測**，開發階段移動段落以「定點轉頭 + 舉手」替身排練。

#### 5. 多機（核心機制，最高風險）

- **方案 A（優先，風險可控）**：**自寫 socket**。UDP 送已實測，先用「中控 UDP 廣播 cue → 各站收 → 回 ack」；訊息格式 `{from, role, cue, state, ts}`。UDP 收未驗，需先補「雙機收送實測」。若要可靠交棒握手，改 TCP（中控當 server、各站當 client）保證 ack 不漏。
- **方案 B（備援）**：非公開 `ConnectionManager`（WiFi WebSocket P2P），但 Unity glue 不完整（**缺 startListen/send**），需**補 Java glue** 才能用；列為次選，僅在 socket 不穩時投入。
- **前置阻塞項（必做）**：⚠️**從未雙機實跑**。上線前先完成「**雙機收送實測**」：兩台同網段，一台送一台收，確認延遲、丟包、ack 往返；通過後才擴到 3+ 台（≥3 台實體機可用）。
- **多機降級**：見第 7 節 —— 任一台離線，中控自動跳過該站，其餘照走。

#### 6. 必做真機實測（上線前阻塞清單）

| # | 未知能力 | 怎麼測（一句話） |
|---|---|---|
| 1 | **多機 socket 收送**（最高風險） | 兩台同網段，中控 UDP 廣播一則 cue，夥伴機收到後回 ack，確認雙向通＋延遲，再擴到 3 台 |
| 2 | **移動命令是否被授權牆擋** | 在輪式機下 `move(0.1)` 與 `turn(10)`，看是否真的動（會動＝沒被擋，靜默不動＝被擋，啟動移動降級） |
| 3 | **相機權限**（選配視覺） | 在 Kebbi OS 上用 Unity WebCamTexture 開相機抓一幀，確認 OS 是否准一般 app 開相機 |
| 4 | **DOA 解析度/正後方/非語音** | 真機從 0/90/180/270 度發聲，讀 `getDirectionOfDOA()` 看回傳是否準、正後方與非語音聲音是否更新 |
| 5 | **走位開迴路重現性** | 輪式機在膠帶舞台跑同一段「速度×時間」腳本多次，量終點誤差，定出排練校時容差 |

> 第 3 項（相機）僅在啟用選配視覺時才阻塞；核心接力不需要它。
> **不需要 G4 搶麥實測**：本路線不自取麥克風 PCM、不做 STT，無「系統持麥能否搶麥」需求，故該未驗點不適用本路線（非遺漏）。

#### 7. 單機 / 降級備案（呼應決賽完整性）

| 失敗的未驗能力 | 降級後仍可 demo 的版本 |
|---|---|
| **多機通訊（socket 收/收 ConnectionManager 都不通）** | 退成**單機「導演機獨演」**：G5 一台用雲端 TTS 主持 + 逐幀 setMotor 演出，三段台詞自己接力；各站機改為各自獨立單機 demo（不接 cue）。彩蛋壞了不影響任何一隊個人分。 |
| **移動命令被授權牆擋** | 取消「走位到中央」，壓軸改為**各機定點同步動作**（同時轉頭 NECK_Z + 雙臂舉手 setMotor），不移動也有「機器人軍團同步」的震撼。 |
| **相機權限被拒** | 砍掉視覺方位，純靠 DOA 聲源 + cue 腳本驅動演出（視覺本就是選配）。 |
| **DOA 解析度太差** | 不做「面向觀眾」精準轉頭，改成預排固定方位的劇場走位（中控按腳本下 cue）。 |
| **雲端 TTS 斷網/延遲** | 預先把所有台詞離線合成成 .wav 打包進 app，現場用 AudioSource 直接播（不臨場呼叫雲端）。 |

> 降級總原則：合體是**疊在各隊獨立作品上的一層**；任一未驗能力失敗，剝掉該層，各站仍能單機完整 demo。

#### 8. 學生分工與難度

| 模組 | 主責隊伍/角色 | 程式深度 | 內容 |
|---|---|---|---|
| **中控狀態機 + 演出腳本** | G5（高三升大學） | 高 | 純 C# scene→cue→next 狀態機、計時計分、交棒握手邏輯 |
| **多機通訊層（socket）** | G5 進階成員 | 最高 | UDP/TCP 自寫送收、訊息格式、ack；雙機收送實測；（備援）補 ConnectionManager Java glue |
| **雲端 TTS + AudioSource** | G5/G2 共用 | 中 | UnityWebRequest 接 Azure/Google、音檔快取、AudioSource 播放、離線 wav 備案（純輸出，不做 STT/不取 PCM） |
| **逐幀動作編排（指向/舉手/勝利姿勢）** | G2（國二升國三）、G3（國三升高一） | 中 | setMotorPositionInDegree 關鍵影格、getMotorPresentPossitionInDegree 回授校正 |
| **DOA 轉頭 workaround** | G3 | 中 | 讀 getDirectionOfDOA → setMotor 轉 NECK_Z（取代 turnToDOA），含 DOA 解析度實測 |
| **壓軸走位（速度×時間）** | G2/G3 各派 1 人 | 中低 | move/turn 開迴路腳本、地面膠帶標記、排練校時；含移動是否被擋實測 |
| **選配視覺（WebCamTexture+MediaPipe）** | G5/G3 行有餘力 | 高 | 自取影像、臉框中心推方位；含相機權限實測 |
| **現場安全/緊急停止** | 每隊各 1 人 | 低 | 各盯自己那台狀態、手持急停 |

> 年齡對應：高難度的通訊/中控/視覺集中在 G5（最年長、程式最深）；G2/G3 負責動作編排與走位這類「邏輯清楚、可逐步排練」的中等模組；安全急停人人可擔任。
